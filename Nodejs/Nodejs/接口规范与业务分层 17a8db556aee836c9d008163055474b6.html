接口规范与业务分层

  • 1、接口设计原则
    • RESTful 风格优先：以资源为中心，使用 HTTP 方法表达操作语义（GET/POST/PUT/DELETE），例如：
      • GET /api/v1/users：查询用户列表
      • POST /api/v1/users：创建用户
      • PUT /api/v1/users/:id：更新指定用户
      • DELETE /api/v1/users/:id：删除指定用户
    • 路径命名规范：使用名词复数（而非动词），全小写 + 连字符，例如/api/v1/user-posts（而非/getUserPosts）。
    • 版本控制：在 URL 中显式声明版本（推荐），如/api/v1/...，便于多版本并行维护。
  • 2、请求规范
    • 参数传递：
      • 路径参数：用于标识资源（如/users/:id中的id）
      • 查询参数：用于过滤 / 分页 / 排序（如/users?page=1&size=10&role=admin）
      • 请求体（Body）：POST/PUT 请求的复杂数据，使用 JSON 格式（Content-Type: application/json）
    • 参数验证：强制验证所有输入（使用 Joi/Zod 等工具），例如校验用户创建接口的name（非空）、age（数字且 > 0）。
  • 3、响应规范

    统一响应格式：所有接口返回结构一致，便于前端统一处理：

    {
  "code": 200,       // 业务状态码（200成功，其他为错误）
  "message": "success", // 状态描述（错误时为具体信息）
  "data": { ... }    // 业务数据（成功时返回，错误时可选）
}
    • HTTP 状态码配合：响应的 HTTP 状态码与业务码对应，例如：
      • 200（OK）：成功
      • 400（Bad Request）：客户端参数错误
      • 401（Unauthorized）：未认证
      • 403（Forbidden）：权限不足
      • 500（Internal Server Error）：服务器异常
  • 4、错误处理规范
    • 错误分类：区分客户端错误（参数 / 权限）和服务器错误，返回具体错误信息（避免暴露敏感细节）。
    • 示例：参数错误响应
    {
  "code": 40001,
  "message": "用户名不能为空",
  "data": null
}
  • 5、其他规范
    • 跨域处理：通过 CORS 中间件统一配置允许的 origin/methods/headers。
    • 接口文档：使用 Swagger/OpenAPI 自动生成文档，包含参数、响应、示例。
    • 幂等性：PUT/DELETE 接口需保证幂等（多次调用结果一致），避免重复创建 / 删除。
  • 业务分层架构
    • 1、路由层（Routes）

      定义 API 路径与控制器的映射，仅负责 "分发请求"，不处理业务逻辑。

      // routes/user.routes.js
const express = require('express');
const router = express.Router();
const userController = require('../controllers/user.controller');

// 映射路径到控制器方法
router.get('/', userController.getUsers);
router.post('/', userController.createUser);
router.put('/:id', userController.updateUser);

module.exports = router;
    • 2、控制器层（Controllers）
      • 职责：接收请求→验证参数→调用服务层→处理响应，是请求与业务逻辑的 "中间协调者"。
      • 特点：依赖请求（req）和响应（res）对象，不直接操作数据库。
      // controllers/user.controller.js
const userService = require('../services/user.service');
const { validateUser } = require('../validators/user.validator');

exports.createUser = async (req, res, next) => {
  try {
    // 1. 验证参数
    const { error, value } = validateUser(req.body);
    if (error) return res.status(400).json({ 
      code: 40001, 
      message: error.details[0].message 
    });

    // 2. 调用服务层处理业务
    const user = await userService.create(value);

    // 3. 返回响应
    res.json({ code: 200, message: '创建成功', data: user });
  } catch (err) {
    next(err); // 交给错误中间件处理
  }
};
    • 3、服务层（Services）
      • 职责：实现核心业务逻辑（如权限校验、数据计算、流程控制），是业务的 "大脑"。
      • 特点：纯函数式设计（不依赖 req/res），仅接收参数并返回结果，便于单元测试。
      // services/user.service.js
const userDal = require('../dal/user.dal');
const { encryptPassword } = require('../utils/crypto');

exports.create = async (userData) => {
  // 1. 业务逻辑：加密密码
  const encryptedPwd = await encryptPassword(userData.password);
  
  // 2. 调用数据访问层保存
  const user = await userDal.create({
    ...userData,
    password: encryptedPwd
  });

  // 3. 脱敏处理（移除敏感字段）
  const { password, ...userWithoutPwd } = user;
  return userWithoutPwd;
};
    • 4、数据访问层（DAL-Data Access Layer）
      • 职责：封装与数据库的交互（查询 / 插入 / 更新），是业务层与数据库的 "隔离层"。
      • 特点：依赖 ORM / 数据库驱动（如 Sequelize/Mongoose），对外暴露简洁接口。
      // dal/user.dal.js
const UserModel = require('../models/user.model');

exports.create = async (data) => {
  return await UserModel.create(data);
};

exports.findById = async (id) => {
  return await UserModel.findByPk(id);
};
    • 5、模型层（Models）

      定义数据结构、校验规则、数据库映射关系（如字段类型、关联关系）。

      // models/user.model.js
const { DataTypes } = require('sequelize');
const sequelize = require('../config/db');

const User = sequelize.define('User', {
  name: {
    type: DataTypes.STRING,
    allowNull: false,
    validate: { notEmpty: true }
  },
  age: {
    type: DataTypes.INTEGER,
    validate: { min: 0 }
  }
});

module.exports = User;
    • 6、公共层（共享资源）
      • 中间件（Middleware）：处理跨切面逻辑（如日志、认证、错误捕获）。
      • 工具函数（Utils）：通用功能（加密、日期格式化、校验）。
      • 配置（Config）：数据库、端口等环境配置。
  • 分层调用关系

    请求流向：客户端 → 路由层 → 控制器层 → 服务层 → 数据访问层 → 数据库

    响应流向：数据库 → 数据访问层 → 服务层 → 控制器层 → 客户端
