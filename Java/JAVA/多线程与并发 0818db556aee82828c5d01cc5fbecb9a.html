多线程与并发

使用多线程可以让程序充分利用CPU的资源，提高CPU的使用效率，从而解决高并发所带来的负载均衡问题，

优点：

  • 资源得到合理利用
  • 程序设计更加简洁
  • 程序响应速度更快，运行效率更高

缺点：

  • 需要更多的内存空间来支持多线程
  • 多线程并发访问的情况可能会影响数据的准确性
  • 数据被多线程共享，可能会出现死锁的情况

应该将程序设计的更加合理有效，避免多线程的缺点，充分发挥多线程的优点，从而提高程序的性能

进程和线程

什么是进程？

进程是计算机正在运行的一个独立的应用程序，进程是一个动态的概念。

什么是线程？

线程是组成进程的基本单位，一个进程中包含一个或多个线程，线程可以完成特定的功能。

进程和线程都是应用程序在执行过程中的概念，动态，如果应用程序没有执行，则不存在进程和1线程
应用程序是静态的概念，进程和线程都是动态的概念，有创建就销毁，存在也是暂时的，不是永久的
进程和线程的区别？
进程在运行时拥有独立的内存空间，每个进程所占用的内存都是独立的，互不干扰。
多个线程时共享空间的，但是每个线程的执行时相对独立的，只不过公用内存空间。
线程必须依赖进程才能执行，单独的线程是无法执行的，由进程来控制多个线程的执行。
多线程：一个进程中，多个线程同时执行。
单核CPU，多线程并不是真正的同时执行，而是多个线程交替占用CPU，执行自己的业务。

线程的状态

线程一共有5种状态，在特定的情况下，线程可以在不同的状态之间切换

  • 创建状态：实例化了一个新的线程对象，还未启动
  • 就绪状态：创建好的线程对象调用了start()方法完成启动，进入线程池等待抢占CPU资源
  • 运行状态：线程对象获取了CPU资源，在规定的时间内执行任务
  • 阻塞状态：正在运行的线程暂时执行任务，释放所占用的CPU资源
  • 终止状态：线程运行完成或因为异常导致线程终止运行

线程调度

线程休眠

休眠是指让当前线程暂停执行，从运行状态进入阻塞状态，从而将CPU资源让给其他线程使用的一种调度方式，通过sleep方法实现(暂时的状态)。

public static void sleep(Duration duration) throws InterruptedException
public class Main {
    public static void main(String[] args) {
        Thread thread  = new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                if ( i==5) { // i == 5 时候休眠3秒钟
                    try {
                        Thread.sleep(3000);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
                System.out.println("---------------------" + i + "-----------------");
            }
        });
        thread.start();
    }
}

对于继承了Thread类的方法，可以直接调用sleep方法，因为Thread作为父类中有sleep方法

sleep(3000);

而如果是使用lambda参数接口，不可以直接调用，只能类调用（sleep是静态方法）

Thread.sleep(3000);

线程合并

合并是指将指定的某个线程加入到当前线程中，合并为一个线程

由两个线程交替执行变为一个线程中两个子线程顺序执行，一个线程执行完毕之后再来执行第二个线程，通过join方法来实现合并。

线程甲和线程乙，线程甲再执行到某个时间点的时候，调用线程乙的join()方法，则表示从当前时间节点开始cpu资源被线程乙独占，线程甲进入阻塞状态，直到线程乙执行完毕，线程甲继续获取cpu资源执行自己的任务。

谁调用join，将谁合并到对方的线程中
public class Main {
    public static void main(String[] args) {
        joinRunnable joinRunnable  = new joinRunnable();
        Thread thread  = new Thread(joinRunnable);
        thread.start();

        for (int i = 0; i < 100; i++) {
            if ( i == 10) { // 10之前可能是交替的，之后先执行joinRunnable线程内的方法
                try {
                    thread.join();
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
            System.out.println("---------------------" + i + "-----------------");
        }
    }
}

线程同步

多线程同时访问同一个资源，可能会导致数据不准确的情况出现，所以需要通过线程同步来解决

public class Account implements Runnable{
    private  static int num;

    @Override
    public void run () {
        num ++;
        try {
            Thread.sleep(1);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        System.out.println(Thread.currentThread().getName() + "是当前的第" + num + "位访客");
    }
}
加入线程1先抢到cpu资源，这时侯执行num++, num变为1，然后线程1休眠
1休眠后，2获得资源，num++,num变为2，然后2休眠，
之后1先醒过来，执行输出，此时输出num=2
随后2醒过来，执行输出，结果仍然是num=2
| 如果没有休眠，结果也有可能是错误的，因为cpu资源不一定什么时候交换，可能num++后就交换，导致结果错误

        t2.start();

关键在于synchronized 锁的是什么

判断多个线程是否会实现同步，看锁定的资源再内存中有几份，如果只有一份，多线程肯定要排队，如果有多份，则多线程不需要排队，就不会实现同步

synchronized 还可以修饰代码块，会为代码块加上锁，从而实现同步

包装类常量池有一个可用范围 -128~127，在此区间内常量池可用，同一个数值可以给多个线程使用，超出返回则失效，此时需要单独在堆中创建对应的数据，不是一个对象而是多个对象。

死锁

死锁：多个线程因为争夺同一个资源而形成的一种互斥的状态，导致每个线程都无法继续进行，程序一直处于卡顿状态。

例如
A 拿一根筷子1，试图获取筷子2
B 拿一根筷子2，试图获取筷子1
两者谁都不愿意舍弃自己的筷子，导致无法下一步

程序模拟

public class DeadLockRunnable implements Runnable {
	public int num;
	private static Data data1 = new Data();
	private static Data data2 = new Data();
	
	@Override
	public void run() {
		if (num == 1) {
			System.out.printlne(Thread.currentThread().getName() + "获取1，等待2");
			synchronized (data1) {
				try {
					Thread.sleep(100);
				} catch (InterruptedException e) {
					e.printStackTrace();
				}
				
				synchronized (data2) {
					System.out.printlne(Thread.currentThread().getName() + "获得2，用餐完毕");
				}
			}
		}
		
		if (num == 2) {
			System.out.printlne(Thread.currentThread().getName() + "获取2，等待1");
			synchronized (data2) {
				try {
					Thread.sleep(100);
				} catch (InterruptedException e) {
					e.printStackTrace();
				}

					System.out.printlne(Thread.currentThread().getName() + "获得1，用餐完毕");

		}
	}
}

重入锁

ReentrantLock 是对 synchronized的升级，synchronized 是通过JVM实现，ReentrantLock是通过JDK实现（用方法调用的形式来使用）

重入锁的特点是可以重复上锁

synchronized 是自动上锁，自动解锁，ReentrantLock是手动上锁，手动解锁

import java.util.concurrent.locks.ReentrantLock;

public class Account implements Runnable{
    private  static int num;
    private ReentrantLock reentrantLock  = new ReentrantLock();

    @Override
    public void run () {
        reentrantLock.lock();
        reentrantLock.lock();
        num ++;
        try {
            Thread.sleep(1);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        System.out.println(Thread.currentThread().getName() + "是当前的第" + num + "位访客");
        reentrantLock.unlock();
        reentrantLock.unlock(); // 上面锁几次，下面就要开锁几次，否则还是锁者的状态
    }
}

ReentrantLock 具有限时性，指可以判断某个线程在一定时间内能否可以拿到锁（tryLock(long timeout, TimeUnit unit)）

import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.ReentrantLock;

public class TiemLock  implements Runnable{
    private ReentrantLock reentrantLock = new ReentrantLock();

    @Override
    public void run() {
        try {
            if (reentrantLock.tryLock(3, TimeUnit.SECONDS)) {
                System.out.println(Thread.currentThread().getName() + "获取到了锁");
                Thread.sleep(4000);
            }
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally  {
		        if (reentrantLock.isHeldByCurrentThread())
	            reentrantLock.unlock();
        }
    }
}
// isHeldByCurrentThread() 锁是否是当前线程所拥有的
ReentrantLock 比 synchronized 更加灵活

生产者消费模式

多线程协作模式，用于解决两个或多个线程之间数据生产于消费的同步问题。
该模式通过一个共享缓冲区（“队列”）来解耦生产者和消费者，生产者负责生成数据并放入缓冲区，消费者从缓冲区取出数据并进行处理。

当缓冲区满时生产者等待，缓冲区空时消费者等待。

需要确保生产者和消费者对缓冲区的互斥访问，以及条件等待（满则等、空则等）。

如何实现？

  1. 使用 synchronized + wait() / notifyAll()

生产者满时调用wait() ，消费者空时调用wait() 生产/消费后调用notifyAll() 唤醒所有等待线程

  • 代码示例
    import java.util.LinkedList;
import java.util.Queue;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

public class ProducerConsumerLock {
    private static final int CAPACITY = 5;
    private final Queue<Integer> buffer = new LinkedList<>();
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    class Producer implements Runnable {
        @Override
        public void run() {
            int value = 0;
            while (true) {
                lock.lock();
                try {
                    while (buffer.size() == CAPACITY) {
                        notFull.await();
                    }
                    buffer.offer(value);
                    System.out.println("Produced: " + value);
                    value++;
                    notEmpty.signal(); // 唤醒一个消费者
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                } finally {
                    lock.unlock();
                }
                try { Thread.sleep(500); } catch (InterruptedException e) { break; }
            }
        }
    }

    class Consumer implements Runnable {
        @Override
        public void run() {
            while (true) {
                lock.lock();
                try {
                    while (buffer.isEmpty()) {
                        notEmpty.await();
                    }
                    int value = buffer.poll();
                    System.out.println("Consumed: " + value);
                    notFull.signal(); // 唤醒一个生产者
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                } finally {
                    lock.unlock();
                }
                try { Thread.sleep(1000); } catch (InterruptedException e) { break; }
            }
        }
    }

    public static void main(String[] args) {
        ProducerConsumerLock demo = new ProducerConsumerLock();
        Thread producer = new Thread(demo.new Producer());
        Thread consumer = new Thread(demo.new Consumer());
        producer.start();
        consumer.start();
    }
}
  • 注意
    • 必须使用 while 循环检查条件，而非 if，避免虚假唤醒（spurious wakeup）。
    • notifyAll() 会唤醒所有等待线程，可能造成性能浪费，但比 notify() 更安全（避免只唤醒同类线程导致死锁）。
  1. 使用 ReentrantLock + Condition

Lock 接口提供了更灵活的锁机制，Condition 可以分别管理生产者等待和消费者等待

优势：可以精确唤醒生产者或消费者，避免 notifyAll() 带来的性能损耗。

  • 代码示例
    ```java
import java.util.LinkedList;
import java.util.Queue;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

public class ProducerConsumerLock {
    private static final int CAPACITY = 5;
    private final Queue<Integer> buffer = new LinkedList<>();
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    class Producer implements Runnable {
        @Override
        public void run() {
            int value = 0;
            while (true) {
                lock.lock();
                try {
                    while (buffer.size() == CAPACITY) {
                        notFull.await();
                    }
                    buffer.offer(value);
                    System.out.println("Produced: " + value);
                    value++;
                    notEmpty.signal(); // 唤醒一个消费者
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                } finally {
                    lock.unlock();
                }
                try { Thread.sleep(500); } catch (InterruptedException e) { break; }
            }
        }
    }

    class Consumer implements Runnable {
        @Override
        public void run() {
            while (true) {
                lock.lock();
                try {
                    while (buffer.isEmpty()) {
                        notEmpty.await();
                    }
                    int value = buffer.poll();
                    System.out.println("Consumed: " + value);
                    notFull.signal(); // 唤醒一个生产者
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                } finally {
                    lock.unlock();
                }
                try { Thread.sleep(1000); } catch (InterruptedException e) { break; }
            }
        }
    }

    public static void main(String[] args) {
        ProducerConsumerLock demo = new ProducerConsumerLock();
        Thread producer = new Thread(demo.new Producer());
        Thread consumer = new Thread(demo.new Consumer());
        producer.start();
        consumer.start();
    }
}
```

Sleep和Wait的区别

sleep和wait的功能类似，都是让线程暂停执行任务

sleep是 Thread类 提供的方法，wait是Object类提供的方法

sleep是直接作用于线程对象本身，wait作用于线程正在访问的资源

调用A对象的wait：让当前正在访问A对象的线程休眠，同时它有一个前提，当线程必须拥有A对象，所以wait方法只能在同步(synchronized)方法或同步代码块中调用，否则会有异常
wait会释放锁，sleep不会释放锁
import java.util.concurrent.TimeUnit;

}

Synchronized锁定的是谁

如果修饰非静态方法，则锁的是方法的调用者

当 synchronized 修饰静态方法和实例方法时，它们使用的锁是不同的：

  • 静态方法：锁的是类的 Class 对象（全局唯一）。
  • 实例方法：锁的是当前实例对象（每个实例有自己的锁）。

因此，静态同步方法与实例同步方法不会互相阻塞，因为锁的对象不同。下面通过一个简单例子演示这一点

  • 例子
    public class SyncDemo {

    // 静态同步方法，锁住 SyncDemo.class
    public static synchronized void staticMethod() {
        System.out.println(Thread.currentThread().getName() + " 进入静态方法");
        try {
            Thread.sleep(2000); // 模拟耗时操作
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println(Thread.currentThread().getName() + " 离开静态方法");
    }

    // 实例同步方法，锁住当前对象 this
    public synchronized void instanceMethod() {
        System.out.println(Thread.currentThread().getName() + " 进入实例方法");
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println(Thread.currentThread().getName() + " 离开实例方法");
    }

    public static void main(String[] args) {
        SyncDemo obj = new SyncDemo();

        // 线程1：调用静态方法
        Thread t1 = new Thread(() -> {
            SyncDemo.staticMethod();
        }, "Thread-1");

        // 线程2：调用同一个对象的实例方法
        Thread t2 = new Thread(() -> {
            obj.instanceMethod();
        }, "Thread-2");

        t1.start();
        t2.start();
    }
}

    解释：

    • staticMethod() 使用类锁（SyncDemo.class）。
    • instanceMethod() 使用对象锁（obj 实例）。
    • 两个锁互不干扰，因此线程1和线程2可以并发执行各自的方法。

首先判断方法是否添加了synchronized关键：

如果没有添加，则不需要考虑线程同步的问题，

如果添加了，则需要考虑线程同步的问题。

看当前锁定的资源在内存中有几份：

如果只有一份，则多个线程会同步，

如果有多份，则多个线程不需要同步。

    public boolean add(E e) {
	final ReentrantLock lock = this.lock;
	lock.lock();
	try {
		Object[] elements = getArray();
		int len = elements.length;
		Object[] newElements = Arrays.copyOf(elements, len+1);
		newElements[len] = e;
		return true;
	} finally {
		lock.unlock();
	}
}

    可以对CopyWrite容器进行并发的读，而不需要加锁，因为当前容器不会添加任何元素，添加元素都是针对复制出来的新集合进行操作，所以CopyOnWrite容器也是一种读写分离的死想，读和写操作的 是不同的容器。

JUC并发编程工具包

CountDownLatch

减法计数器，JUC的工具类，可以用来倒计时，当两个线程同时执行时候，如果我们要确保一个线程优先执行。设置一个计数器，当计数器清零的时候再执行另一个。

import java.util.concurrent.CountDownLatch;
public class Main {
    public static void main(String[] args) {
		    // 计数器为100，即为执行100次计数
        CountDownLatch countDownLatch = new CountDownLatch(100); 
        new Thread(() -> {
            for (int i = 0; i < 100; i++) {
                System.out.println("thread-----" + i);
                countDownLatch.countDown();
            }
        }).start();

        try {
            // 计数器等待，计数器才会起作用
            countDownLatch.await();
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }

        for (int i = 0; i < 100; i++) {
            System.out.println("main+++++" + i);
        }
    }
}
如果计数器上的计数小于实际次数，再计数内是正常的，之后就是交替争夺cpu资源状态
如果i计数器上的计数大于实际次数，导致计数无法减为0，实际次数执行完毕后，程序阻塞

CyclicBarrier

加法计数器，从0开始加，与 CountDownLatch 相反

当调回线程执行指定次数后，执行第二个参数的方法（第二个参数为函数）

import java.util.concurrent.*;
public class Main {

读写锁

接口 ReadWriteLock， 实现类 ReentranReadWriteLock，可以多线程同时读，但是同一时间只能有一个进程操作。

读写锁功能也是为了实现线程同步，只不过粒度更精细，可以分贝给读和写操作设置不同的锁机制。

public class Main {
    public static void main(String[] args) {
        Cache cache = new Cache();
        for (int i = 0; i < 5; i++) {
            final int temp = i;
            new Thread(() -> {
                cache.write(temp, String.valueOf(temp));
            }).start();
        }

        for (int i = 0; i < 5; i++) {
            final int temp = i;
            new Thread(() -> {
                cache.read(temp);
            }).start();
        }
    }
}

class Cache {
    private Map<Integer, String> map= new  HashMap<> ();
    ReadWriteLock lock = new ReentrantReadWriteLock();


    // 写操作
    public void write(Integer key, String value) {
        lock.writeLock().lock(); // 添加写锁
        System.out.println(key + "--开始写入");
        map.put(key, value);
        System.out.println(key + "--写入完毕");
        lock.writeLock().unlock(); // 写完解除锁
    }

    // 读操作
    public void read( Integer key) {
        lock.readLock().lock(); // 读锁
        System.out.println(key + "开始读取");
        map.get(key);
        System.out.println(key + "读取结束");
        lock.readLock().unlock(); // 解除
    }
}

线程池

线程池是一种缓冲机制，预先创建好一定数量的线程对象，存入缓冲池，需要用的时候直接从缓冲池中取出来，用完之后不销毁，还回到线程池中，供下一次任务使用。

优点：可以提高线程的利用率，提高响应速度，便于统一管理线程对象，可以控制最大并发量

工作流程：

  • 当任务过多的时候，额外补充线程数量
  • 任务趋于正常时候，额外补充的线程会自动销毁

初始化线程数量

最大线程数量（任务过多的时候开启，任务少的时候销毁）

ForkJoin框架

多线程并发处理框架（JDK1.7 之后），本质上是对线程池的一种补充，

核心思想是将一个大型任务拆分成多个小任务，小任务处理后结果合并汇总成总结果。

本质：多个任务 + 多个线程
  • ForkJoinTask 表示任务
  • ForkJoinPool 表示线程（线程池的一种扩展）

递归

什么是递归？ 函数直接或间接调用函数本身。递推公式

递归需要满足三个条件：

  1. 一个父问题可以拆分为若干个子问题，并且若干个子问题的结果汇总起来就是父问题的结果
  1. 父亲问题和子问题，解题思路完全一致，只是数据规模不同
  1. 存在终止条件
  1. 创建一个ForkJoinTask任务，ForkJoinTask是一个抽象类，需要创建一个类来继承ForkJoinTask子子RecursiveTask，实现抽象方法compute，拆分的逻辑就写在compute方法中。
  1. 任务要通过ForkJoinPool来执行，将任务直接放入ForkJoinPool中，直接获取结果即可。

比如：计算0 ~ 20亿数字相加之和 (框架使用)

import java.util.concurrent.RecursiveTask;

public class ForkJoinDemo extends RecursiveTask<Long> {
    private final long start;
    private final long end;
    private static final long THRESHOLD = 1_000_000L;

    public ForkJoinDemo(long start, long end) {
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0L;
            for (long i = start; i < end; i++) {
                sum += i;
            }
            return sum;
        } else {
            long mid = (start + end) >>> 1; // 使用无符号右移防止溢出
            ForkJoinDemo left = new ForkJoinDemo(start, mid);
            ForkJoinDemo right = new ForkJoinDemo(mid, end); // 左闭右开，mid 交给右任务
            left.fork();// 将左子任务异步提交到线程池
            return right.compute() + left.join();// 当前线程直接执行右子任务，然后等待左子任务结果
        }
    }
}
public class Main {
    public static void main(String[] args) {
        Long  sum = 0L;
        Long startTime = System.currentTimeMillis();
//        for (int i = 0; i < 20_0000_0000L; i++) {
//            sum += i;
//        }
        ForkJoinPool forkJoinPool = new ForkJoinPool();
        ForkJoinTask<Long> task = new ForkJoinDemo(0L, 20_0000_0000L);
        forkJoinPool.execute(task); // 结果比直接使用for快百倍
        try {
            sum = task.get();
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        } catch (ExecutionException e) {
            throw new RuntimeException(e);
        }
        Long endTime = System.currentTimeMillis();
        System.out.println(sum + "总耗时 = " + (endTime - startTime) + "mm");
    }
}

Volatile关键字

是一种由Java虚拟机提供的轻量级同步机制，保证共享变量的可见性和有序性，但不保证原子性

当一个线程修改了 volatile 变量的值，新值会立即被写回主内存，其他线程读取该变量时，会从主内存重新读取，而不是使用自己工作内存中的缓存副本。

Java内存模型（JMM）规定，每个线程都由自己的工作内存（可以视作cpu缓存），线程对变量的操作必须是先在自己的工作内存中进行，然后再刷新到主内存。
普通变量再不同线程之间可能由于缓存不一致而大致数据不一致。

单例模式的写法(双重检查锁（Double-Checked Locking）单例模式)

public class SingletonDemo {
    private volatile static SingletonDemo instance;
    public SingletonDemo() {
        System.out.println("创建了一个SingletonDemo对象");
    }
    public static SingletonDemo getInstance() {
        if (instance == null) {
            synchronized (SingletonDemo.class) {
                if (instance == null) {
                    instance = new SingletonDemo();
                }
            }
        }
        return instance;
    }
}

instance = new Singleton(); 实际包含三个步骤：
1. 分配内存空间
2. 初始化对象（调用构造器）
3. 将内存地址赋值给 instance 引用

volatile关键字的作用是可以使内存中的数据对线程可见

Java内存模型JMM （Java Memory Model）

一个线程在访问内存数据的时候，其实不是拿到数据本身，而是将数据复制保存到工作内存中，相当于使用的是一个副本，对工作内存中的数据进行修改，修改完成之后再保存到主内存中，主内存对线程不可见。

1. 禁止优化：不要以为这个值没变就不去读它。
2. 直接读内存：每次使用这个变量，都必须去主内存里取最新值，不能用缓存里的旧值。
3. 顺序保证：它告诉编译器，这行代码的操作顺序绝对不能乱（不能为了效率把读写顺序打乱，这在多线程里很重要）。
