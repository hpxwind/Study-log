文件类型

TypeScript（TS）的文件类型围绕「源代码编写、类型声明、编译配置、模块系统」四大核心场景设计，不同后缀的文件承担不同职责，且与 TS 编译器（tsc）、模块系统（ES Module/CommonJS）、开发场景（如 React、Node.js）深度绑定。

  • .ts

    TypeScript 基础源文件（最常用）

    TS 的默认源文件，用于编写非 React 场景的普通代码（如 Node.js 后端、工具函数、普通前端逻辑），支持 TS 所有核心特性（类型注解、泛型、映射类型等）。

    • 支持 TypeScript 完整语法（类型注解、接口、枚举、模块导出 / 导入等）；
    • 不支持 JSX 语法（若写 JSX 会报错，需用 .tsx）；
    • 模块系统可通过 tsconfig.json 配置（默认根据文件后缀或 module 选项自动识别）。
    // utils.ts（普通工具函数）
interface User {
  name: string;
  age: number;
}

// 类型注解 + 函数导出
export function formatUser(user: User): string {
  return `姓名：${user.name}，年龄：${user.age}`;
}

    默认编译为 .js 文件（如 utils.ts → utils.js），模块系统由 tsconfig.json 的 module 选项决定（如 CommonJS、ESNext）。

  • .tsx

    支持 JSX 的 TypeScript 源文件（React 专属）

    专门用于编写 React 组件（或其他支持 JSX 的框架，如 Vue 3 + JSX），同时支持 TS 类型特性和 JSX 语法。

    • 完全兼容 .ts 的所有 TS 特性；
    • 支持 JSX 语法（如 <div />、组件渲染），TS 会对 JSX 进行类型检查（如组件属性类型校验）；
    • 特殊规则：JSX.Element 是 JSX 表达式的默认返回类型；需引入 React（或通过 jsxImportSource 配置其他库）才能使用 JSX。
    // UserComponent.tsx
import React from 'react';

// 组件属性类型定义
interface UserComponentProps {
  user: User; // 复用上面定义的 User 接口
  onClick: (name: string) => void;
}

// React 函数组件 + JSX + 类型校验
export const UserComponent: React.FC<UserComponentProps> = ({ user, onClick }) => {
  return (
    <div onClick={() => onClick(user.name)}>
      <p>{formatUser(user)}</p>
    </div>
  );
};

    编译为 .jsx 或 .js 文件（由 tsconfig.json 的 jsx 选项决定，如 react-jsx 模式输出 .js）。

  • .cts

    CommonJS 模块的 TypeScript 源文件（Node.js 专属）

    专门用于 Node.js 环境，明确使用 CommonJS 模块系统（require/module.exports）的 TS 源文件，解决 TS 中模块系统自动识别的歧义。

    • 支持 TS 完整语法；
    • 模块系统固定为 CommonJS，无需在 tsconfig.json 中配置 module；
    • 编译后输出 .cjs 文件（明确标识为 CommonJS 模块，Node.js 可直接运行）。
    // node-utils.cts
// CommonJS 导入（可省略 .ts 后缀）
const { formatUser } = require('./utils');

// CommonJS 导出
module.exports = {
  logUser: (user: User) => {
    console.log(formatUser(user));
  }
};

    node-utils.cts → node-utils.cjs（Node.js 直接执行：node node-utils.cjs）。

  • .mts

    ES Module 模块的 TypeScript 源文件（通用）

    明确使用 ES Module 模块系统（import/export）的 TS 源文件，适用于浏览器、Node.js 14+（支持 ESM）等场景，避免模块系统混淆。

    • 支持 TS 完整语法；
    • 模块系统固定为 ES Module，与 .ts 的默认 ESM 模式一致，但通过后缀明确标识；
    • 编译后输出 .mjs 文件（明确标识为 ESM 模块，需 Node.js 启用 ESM 或浏览器直接引入）。
    // esm-utils.mts
// ES Module 导入
import { formatUser } from './utils.ts';

// ES Module 导出
export const logUser = (user: User) => {
  console.log(formatUser(user));
};

    esm-utils.mts → esm-utils.mjs（Node.js 执行需配置 package.json 的 type: "module"）。

类型声明文件（.d.ts）

类型声明文件（Declaration File）的核心作用是「提供类型信息，不包含具体实现代码」，用于：

  1. 给 JS 文件添加类型提示；
  1. 给第三方 JS 库（无 TS 类型）提供类型支持；
  1. 导出项目的公共类型，供其他模块复用。

TS 编译器会自动识别 .d.ts 系列文件，且不会将其编译为 JS 文件（仅用于类型检查）。

  • 基础声明文件 .d.ts

    通用类型声明文件，适用于大多数场景（如项目公共类型、JS 库类型补充）。

    • 仅包含类型定义（接口、类型别名、枚举、模块声明等），无具体代码逻辑（如函数实现、变量赋值）；
    • 支持 declare 关键字（声明全局类型、模块类型、第三方库类型）。

    1.项目公共类型到处

    // types.d.ts（项目公共类型）
export interface User {
  name: string;
  age: number;
  address?: {
    province: string;
    city: string;
  };
}

export type Gender = 'male' | 'female' | 'other';

    其他文件可直接导入使用：

    import { User, Gender } from './types';
const user: User = { name: '张三', age: 20 };

    2.给JS文件天机啊类型声明

    若项目中存在 .js 文件，可通过同名 .d.ts 文件提供类型：

    // utils.js（JS 文件，无类型）
export function formatUser(user) {
  return `姓名：${user.name}，年龄：${user.age}`;
}
    // utils.d.ts（对应 JS 文件的类型声明）
export interface User {
  name: string;
  age: number;
}

export declare function formatUser(user: User): string;

    3.全局类型声明

    若需在所有文件中直接使用类型（无需导入），可声明全局：

    // global.d.ts（全局类型声明）
declare global {
  interface Window {
    // 扩展 Window 对象的类型
    appConfig: {
      apiUrl: string;
      timeout: number;
    };
  }

  type JsonValue = string | number | boolean | null | object;
}

// 无需导出，全局可访问

    其他文件直接使用：

    console.log(window.appConfig.apiUrl); // 类型提示正常
const data: JsonValue = { name: '张三' }; // 全局类型

    4.第三方JS库类型声明（无@types包时）

    若使用的 JS 库（如 xxx-lib）无官方类型，可手动声明：

    // xxx-lib.d.ts（第三方库类型声明）
declare module 'xxx-lib' {
  export function doSomething(arg: string): void;
  export const version: string;
}

    导入使用时会有类型提示：

    import { doSomething } from 'xxx-lib';
doSomething('test'); // 类型提示：arg 为 string
  • 模块专属声明文件 .d.cts / .d.mts

    分别对应 .cts（CommonJS）和 .mts（ES Module）的类型声明文件，明确模块系统的类型适配。

    // node-utils.d.cts（对应 node-utils.cts 的类型声明）
import type { User } from './types';
declare function logUser(user: User): void;
export = logUser; // CommonJS 风格导出类型
    // esm-utils.d.mts（对应 esm-utils.mts 的类型声明）
import type { User } from './types';
export declare function logUser(user: User): void; // ESM 风格导出类型
  • 自动生成声明文件

    无需手动编写 .d.ts，可通过 tsconfig.json 配置让 TS 编译器自动生成：

    // tsconfig.json
{
  "compilerOptions": {
    "declaration": true, // 编译时自动生成 .d.ts 文件
    "declarationDir": "./types", // 声明文件输出目录
    "declarationMap": true // 生成 .d.ts.map 源映射文件（方便调试）
  }
}

    编译后，utils.ts 会自动生成 utils.d.ts，包含所有导出类型的声明。

  • 第三方库类型声明 @type/xxx

    大多数流行 JS 库（如 React、Vue、Node.js）的类型声明已由社区维护在 @types 仓库中，无需手动编写，直接安装即可

    # 安装 React 的类型声明
npm install @types/react --save-dev

# 安装 Node.js 的类型声明
npm install @types/node --save-dev

TS配置文件（tsconfig.json 系列）

tsconfig.json 是 TS 项目的核心配置文件，用于指定编译选项、文件包含 / 排除规则、模块系统、目标环境等。TS 编译器会自动查找项目根目录的 tsconfig.json，按配置执行编译和类型检查。

  • 核心配置文件：tsconfig.json

    项目默认配置文件，适用于大多数场景（前端、后端、全栈项目）。

    {
  "compilerOptions": {
    // 目标 JS 版本（如 ES5、ES6、ESNext）
    "target": "ESNext",
    // 模块系统（CommonJS、ESNext、UMD 等）
    "module": "ESNext",
    // 模块解析策略（Node、Classic）
    "moduleResolution": "Node",
    // 输出目录（编译后的 JS 文件存放位置）
    "outDir": "./dist",
    // 源文件目录（TS 源文件存放位置）
    "rootDir": "./src",
    // 是否自动生成 .d.ts 声明文件
    "declaration": true,
    // JSX 处理模式（react、react-jsx、preserve 等）
    "jsx": "react-jsx",
    // 严格模式（开启所有严格类型检查，推荐启用）
    "strict": true,
    // 允许导入 JS 文件
    "allowJs": true,
    // 检查 JS 文件的类型（需配合 allowJs 使用）
    "checkJs": true,
    // 忽略类型声明文件的错误
    "skipLibCheck": true,
    // 模块导入时是否允许省略后缀（如 .ts、.tsx）
    "moduleSuffixes": [".ts", ".tsx", ".js", ".jsx"]
  },
  // 需编译的文件（支持 glob 模式）
  "include": ["./src/**/*"],
  // 排除的文件
  "exclude": ["./node_modules", "./dist"],
  // 继承其他配置文件（如社区通用配置、子配置）
  "extends": "./tsconfig.base.json"
}

    作用：

    • 统一项目的编译规则，避免不同开发者的编译差异；
    • 控制 TS 类型检查的严格程度（如 strict: true 开启严格模式，减少类型隐患）；
    • 指定输出目录、声明文件生成等，规范项目结构。
  • 子配置文件：tsconfig.node.json / tsconfig.app.json

    适用于 monorepo 项目（如前端 + 后端、主应用 + 子应用），或需要区分不同环境（如开发 / 生产、Node.js/ 浏览器）的项目，用于拆分不同的配置规则。

    示例：（React+Node.js 全站项目）

    • tsconfig.app.json：前端 React 应用配置（处理 .tsx、JSX、浏览器环境）；
    • tsconfig.node.json：Node.js 后端配置（处理 .cts、CommonJS、Node.js 环境）。
    // tsconfig.node.json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "outDir": "./dist/node",
    "lib": ["ES2020"], // 适配 Node.js 环境的库
    "types": ["node"] // 引入 Node.js 类型声明
  },
  "include": ["./src/node/**/*"],
  "exclude": ["./src/app/**/*"]
}
  • 编译缓存文件：tsconfig.tsbuildinfo

    TS 增量编译的缓存文件，由编译器自动生成和更新，用于加速后续编译（避免重复编译未修改的文件）。

    特点：

    • 无需手动修改，编译器自动管理；
    • 可加入 .gitignore（无需提交到代码仓库）；
    • 启用增量编译需在 tsconfig.json 中配置：
    "compilerOptions": {
  "incremental": true, // 开启增量编译
  "tsBuildInfoFile": "./tsconfig.tsbuildinfo" // 缓存文件路径
}
文件类型默认模块系统编译输出适用场景
.tsES Module.js浏览器、通用场景
.ctsCommonJS.cjsNode.js 环境
.mtsES Module.mjs明确 ESM 场景（浏览器 / Node.js 14+）
.tsxES Module.js/.jsxReact 等 JSX 框架

若项目使用 CommonJS（Node.js），优先用 .cts 或配置 tsconfig.json 的 module: "CommonJS"；

若使用 ESM，优先用 .mts 或配置 module: "ESNext" + package.json 的 type: "module"；

避免混用不同模块系统的文件（如 .cts 导入 .mts），可能导致编译错误。
