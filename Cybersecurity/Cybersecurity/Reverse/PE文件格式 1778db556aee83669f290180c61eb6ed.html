PE文件格式

PE文件为Windows下的可执行文件。分：PE32，PE+（即64位，PE32+，为32位的扩展形式）

  • PE文件种类

    可执行系列：EXE、SCR

    库系列：DLL、OCX、CPL、DRV

    驱动系列：SYS、VXD

    对象文件系列：OBJ

下面以notepad.exe为例子，研究PE文件基本结构，

  • EXE文件结构（无比重要）
  • VA && RVA && FOA

    VA指 进程虚拟内存的绝对地址

    RVA指 相对虚拟地址，即：从某个基准位置（ImageBase）开始的相对地址

    关系如下：RVA+ImageBase = VA

    FOA指 文件地址偏移

    FOV+FileBuff=导出表地址

    PE头内信息大多以RVA形式存在，为什么？

    >>>PE文件（主要为DLL）加载到进程虚拟内存的特定位置上，该位置可能已经加载了其他PE文件，此时必须重定位（Relocation）到其他空白位置上，否则就会出错。（类比于基地址和偏移地址）

  • PE头

    PE头由许多 结构体 组成。

    • DOS头

      成员说明：

      • 1、标签结构体（Signature）：固定值为0x4550（即：“PE”）

      • 2、文件头（File Header）

        文件头是表现文件大致属性的IMAGE_FILE_HEADER结构体

        typedef struct _IMAGE_FILE_HEADER {
WORD Machine; // 程序运行平台 // 程序允许运行的CPU型号，为0表示可以在任何设备上运行 0x14c 386.....
WORD NumberOfSections; // PE 中节的数量 // 区段数量
DWORD TimeDateStamp; // 时间戳，文件创建日期和时间
DWORD PointerToSymbolTable; // COFF 符号表的文件偏移
DWORD NumberOfSymbols; // 符号表中的符号数量
WORD SizeOfOptionalHeader; // 扩展/可选 PE 头的大小 32bit 默认E0，64bit 默认F0
WORD Characteristics; // PE 文件属性（可读，可写，可执行）
} IMAGE_FILE_HEADER, *PIMAGE_FILE_HEADER;
        • 有四个重要成员（如果设置不正确，文件无法运行）

          1、Machine：每个CPU都有唯一的Machine码，兼容32位Intel x86芯片的Machine码为 14c

          2、NumberOfSections：用来指出文件中存在的节区数量。该值一定要大于0，如果定义的节区数量与实际不符，会发生错误。

          3、SizeOfOptionalHeader：IMAGE_NT_HEADER结构体最后一个成员为IMAGE_OPTIONAL_HEADER32结构体，其用来指出这个结构体的长度。

          （IMAGE_OPTIONAL_HEADER32结构体 由C语言编写而成，大小是确定的，但是Windows的PE装载器需要查看SizeOfOptionalHeader值来确定大小）

          （同时IMAGE_OPTIONAL_HEADER32与IMAGE_OPTIONAL_HEADER64的尺寸不同，因此需要SizeOfOptionalHeader来确认大小）

          4、Characteristics：用于标识文件的属性，文件是否是可运行的状态，是否为DLL信息，以bit OR形式组合起来的。需要记住：

          #define IMAGE_FILE_EXECUTABLE_IMAGE 0x0002 //文件是executable

          #define IMAGE_FILE_DLL 0x2000 //文件是DLL

          还有一个TimeDateStamp成员，不影响文件运行，用来记录编译器创建此文件的时间（VB，VC++支持）

          1、Magic：32位时，值为10B；64位时，值为20B

          2、AddressOfEntryPoint：持有RVA值，该值指出程序最先执行的代码起始位置。

          3、ImageBase：指出文件优先装入地址。虚拟内存的范围是0~FFFFFFFF(32位)。

          EXE，DLL被装载到用户内存的0~7FFFFFFF中，

          SYS被装载入到内核内存的80000000~FFFFFFFF中。

          一般创建好EXE文件后，其Image的值为00400000，DLL的值为10000000.

          之后把EIP寄存器的值设置为EIP = ImageBase + AddressOfEntryPoint

          4、FileAlignment && SectionAlignment：PE文件的Body部分划分为若干节区

          FileAlignment ：指定了节区在磁盘文件中的最小单位。

          SectionAlignment：指定了节区在内存中的最小单位。

          5、SizeOfImage：指定了PE Image在虚拟内存所占空间大小。磁盘文件或内存的节区大小必定为FileAlignment或SectionAlignment值的整数倍。

          6、SizeOfHeader：指出整个PE头的大小，改值也是FileAlignment的整数倍。

          7、Subsystem：用来区分 系统驱动文件（*.sys）与 普通文件（*.exe,*.dll）

          8、NumberOfRvaAndSizes：用来指定DataDirectory表的个数（IMAGE_OPTIONAL_HEADER32结构体中最后一个成员）

          9、DataDirectory：由IMAGE_DATA_DIRECTORY结构体组成的数组，数组中每一项都有被定义的值。

    • 区段/区节头

      区节中定义了各区节属性。正常会有多个，PE文件中的code（代码），data（数据），resource（资源）等按照属性分类存储在不同节区，一定程度上可以保证程序安全，稳定。

      • 不同区节的权限

        code 执行，读

        data 非执行，读写

        resource 非执行，读

      typedef struct _IMAGE_SECTION_HEADER {
BYTE Name[IMAGE_SIZEOF_SHORT_NAME]; // 节名称 // 跟字符串不一样，不会以0结尾。
union {
DWORD PhysicalAddress; // 物理地址
DWORD VirtualSize; // 虚拟大小 // 内存中未对齐的大小
} Misc; // 区段在内存中的真实大小（未对齐）

      节区头是由IMAGE_SECTION_HEADER结构体组成的的数组，每个结构体对应一个节区。

      • 重要成员：

        1、VirtualSize：内存中节区所占大小

        2、VirtualAddress：内存中节区起始地址（RVA），不带有任何值。

        3、SizeOfRawData：磁盘文件中节区所占大小

        4、PointerToRawData：磁盘文件中节区起始位置，不带有任何值。

        5、Characterisitics：节区属性（bit OR）

    IAT是一种表格，用来记录程序正在使用那些库中的哪些函数。

    • DLL

      DLL（Dynamic Linked Library）动态链接库。对于一个程序，由很多库函数组成，对于这个程序的DLL描述：

      1、不要把库包含到程序中，单独组成DLL文件，需要时调用即可。

      2、内存映射技术使加载后的DLL代码，资源在多个进程中实现共享。

      3、更新库时只要替换相关DLL文件即可，简单易行。

      加载DLL的方式分为：

      1、显式链接：使用DLL时加载，使用完后释放内存

      2、隐式链接：程序开始使一同加载DLL，程序终止时再释放内存

      例子：

      调用CreateFileW()函数时并非直接调用，而是胡哦去01001104地址处的值来实现（所有的API调用均采用这种方式）。

      编译器为什么不直接写 CALL 7C8107F0 ？这样不是更精确码？

      >>> notepad.exe程序会运行再不同的操作系统上，kernel32.dll的版本也各不相同，为了都能够正常访问CreateFileW()函数，编译器准备了保存CreateFileW()的函数实际地址（01001104），并以CALL DWORD PTR DS:[01001104]形式记录。

      也有可能时DLL重定位导致的，这个程序使用了多个dll，比如：a.dll，b.dll，PE先将a.dll装在到ImageBase（100000000）处，到了b.dll，由于已经挂在了a.dll，b.dll只能查找其他空白的地方。

        一个进程中的exe，dll文件如何被加载到内存中的？

        1、创建进程内核对象

        2、解析EXE文件的PE头

        3、映射文件到内存

        4、处理重定位表

        5、加载依赖DLL

        6、初始化程序

        EXE 和 DLL 的加载核心是将磁盘文件按 PE 格式映射到进程虚拟内存，并通过解析 PE 头中的导入表、导出表、重定位表等结构，确保程序能正确执行和相互调用。

        并且一般先加载EXE，之后再加载DLL，所以DLL的地址是随机的，因此通过函数名→函数地址（显式）

        EXE调用动态链接库在内存中与在硬盘中有什么不同？

        在硬盘中：

        DLL是静态的，EXE 无法直接访问其内容。DLL仅仅记录调用所需的DLL名称和函数名/序号，作为加载时的索引。

        在内存中：

        操作系统通过LoadLibraryA加载DLL时，会将其映射到进程的虚拟地址空间，并解析EXE导入表，将 DLL 中导出函数的实际内存地址（VA）填充到 EXE 的 IAT（导入地址表） 中。后续EXE调用DLL时直接通过IAT跳转。

        2、OriginalFirstThunk—INT：INT的地址（RVA） | INT是一个包含导入函数信息（Ordinal，Name）的结构体指针数组。每一个地址值分别指向IMAGE_IMPORT_BY_NAME结构体。

        INT是IMAGE_IMPORT_BY_NAME结构体指针数组。数组的第一个元素指向函数的Ordinal值000F，函数名称为PageSetupDlgW。

        4、FirstThunk—IAT：IAT的地址（RVA）| RVA：12C4，即：RAW：6C4

        IAT的第一个元素值被硬编码为 76324906 即comdlg32.dll

	PIMAGE_IMPORT_DESCRIPTOR pImport =  (PIMAGE_IMPORT_DESCRIPTOR)(RvaToFoa(directory.VirtualAddress) + FileBuff);
	while (pImport->OriginalFirstThunk)
	{
		char* dllName = RvaToFoa(pImport->Name) + FileBuff;
		printf("dll文件名称：%s\n", dllName);
		printf("TimeDateStamp=%d\n",pImport->TimeDateStamp);
		PIMAGE_THUNK_DATA pThunkData =(PIMAGE_THUNK_DATA)(RvaToFoa(pImport->OriginalFirstThunk) + FileBuff);
		while (pThunkData->u1.Function)
		{
			// 判断是否按序号导入
			if (pThunkData->u1.Ordinal & 0x80000000)
			{
				printf("按序号导入：%d\n", pThunkData->u1.Ordinal & 0x7FFFFFFF);
			}
			else
			{
				PIMAGE_IMPORT_BY_NAME importNmae = (PIMAGE_IMPORT_BY_NAME)(RvaToFoa(pThunkData->u1.AddressOfData) + FileBuff);

  • 重定位表

    用于在程序加载到内存中时，进行内存地址的修正，

    为什么要进行内存地址的修正？

    例如有一个t.exe需要调用三个动态链接库（a.dll,b.dll,c.dll）t.exe的ImageBase为400000H，三个dll的ImageBase为1000000H。

    加载进内存时，t.exe直接复制到程序的400000H处，接着加载dll，首先是a.dll，直接复制到1000000H处，到b.dll，只能另寻空间，但是b.dll和c.dll中有些地址是根据ImageBase固定写死的，而且是绝对地址，不是偏移。

    因此对于一个程序，这样被写死的地址有很多，需要一张表来记录，每个PE文件都有一张重定位表（Relocation Table）（由于exe一般是第一个加载到内存中的，所以一般exe没有重定位表，但是有的有）

    位于DataDirectory目录中索引为5的位置。

    内存会分页，以0x1000为单位，并且会进行分块

    typedef struct _IMAGE_BASE_RELOCATION {
    DWORD   VirtualAddress;//RVA
    DWORD   SizeOfBlock;
} IMAGE_BASE_RELOCATION,* PIMAGE_BASE_RELOCATION;
#define IMAGE_SIZEOF_BASE_RELOCATION         8

    由于重定位表的SizeOfBlock大小不确定，新的Block的重定位信息的结构体接着上一个Block4字节对齐后开始，而当出现一个_IMAGE_BASE_RELOCATION结构体的值全为0时，表明重定位表结束。

  • TLS

    TLS（Thread Local Storage）：线程局部储存。主要是为了解决多线程中变量同步的问题。

    如何创建TLS变量：（添加_declspec(thread) 其余不变）

    _declspec(thread) int g_number = 100;

    • 用途

      1、线程A去修改TLS变量时，线程B不会受到影响，因为每个线程都拥有一个TLS变量副本。

      2、安全领域中，TLS常被用于处理反调试，抢占执行等操作。

    • TLS回调函数
      #pragma comment(linker, "/SECTION:.CRT$XLX,RWS")
// 添加节属性：确保.CRT$XLX节有读写权限（关键修复）

#pragma data_seg(".CRT$XLX")
// 创建一个名为.CRT$XLX的自定义数据节（section）。
// .CRT是 C 运行时（C Runtime）相关节的前缀。
// $XLX是节的后缀，用于告诉链接器该节的排序和用途（TLS 回调表必须放在特定位置）。
// 该节会被操作系统识别为 TLS 回调函数表的存储位置。
// 节名的命名有严格规范，$X前缀的节会被链接器归类到 TLS 回调相关的区块。

PIMAGE_TLS_CALLBACK pTLS_CALLBACKs[] = {TLS_CALLBACK, 0};
// 存储 TLS 回调函数的地址列表，以0结尾（类似 C 字符串的\0终止符）。
// TLS_CALLBACK是用户定义的回调函数（需自行实现）。
      // 实例：
#include <windows.h>

// 定义TLS回调函数
VOID NTAPI TLS_CALLBACK(PVOID DllHandle, DWORD Reason, PVOID Reserved) {
    switch (Reason) {
        case DLL_PROCESS_ATTACH:    // 进程启动时
            OutputDebugStringA("TLS: 进程启动\n");
            break;
        case DLL_THREAD_ATTACH:     // 线程创建时
            OutputDebugStringA("TLS: 线程创建\n");
            break;
        case DLL_THREAD_DETACH:     // 线程退出时
            OutputDebugStringA("TLS: 线程退出\n");
            break;
        case DLL_PROCESS_DETACH:    // 进程退出时
            OutputDebugStringA("TLS: 进程退出\n");
            break;
    }
}

// 注册TLS回调函数（必须放在自定义的.CRT$XLX节）
#pragma data_seg(".CRT$XLX")
PIMAGE_TLS_CALLBACK pTLS_CALLBACKs[] = {TLS_CALLBACK, 0};
#pragma data_seg()  // 结束自定义节

// 可选：设置节属性（可读可写）
#pragma comment(linker, "/SECTION:.CRT$XLX,RWS")

int main() {
    OutputDebugStringA("进入main函数\n");
    return 0;
}
  • 内存管理-堆

    每个进程都有自己独立的4G内存空间，高2G操作系统内核使用，低2G用户使用。

    每个进程中我们使用的都是虚拟地址，虚拟机地址到物理地址的转换由操作系统内核完成，因此我们无法在自己的进程当中访问其他进程的内存。

    Windows尽量保证同一份数据，在物理内存中只有一份，分别映射到多个进程中。

    当各个进程所使用的内存数量超出物理内存大小后，操作系统还能够将物理内存中暂时用不到的数据交换到硬盘中。

    Windows是如何管理数据的？

    1、堆：适用来管理大量的小型对象，适用堆管理方式可以非常方便的帮我们管理所需要的内存空间，缺点是效率低，控制不够灵活。

    2、虚拟内存：适合用于管理大型的对象数组或大型的结构数组，适用虚拟内存管理方式有丰富的内存管理接口，可以使我们更加精确的控制内存数据。

    3、文件映射：适合用于管理大型的数据流，以及在同一个机器上的不同进程间共享数据。

    堆在Windows中是作为一个对象来管理的，我们可以创建一个堆对象，之后在其基础上分配内存，销毁内存等。C/C++中的 new malloc 其实也是使用的Windows中的堆 对象（Windows API）来分配空间的。

    当Windows系统创建一个进程后，会为此进程创建一个默认的堆，这个默认堆是不能够被销毁的。

    #include<Windows.h>
#include<iostream>

int main() {
	// 创建一个堆对象
	HANDLE hHeap = HeapCreate(0,0,0);
	//  申请内存
	char* str = (char*)HeapAlloc(hHeap, HEAP_ZERO_MEMORY, 10);
	strcpy_s(str,10,"helloword");
	printf("%s\n", str);
	HeapFree(hHeap, 0, str);
	HeapDestroy(hHeap);
	return 0;
}

    提交（Commit）内存被分配，并且与物理内存进行了映射，进程可以使用了

    Image从程序的PE映像映射而来
