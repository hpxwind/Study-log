周期执行代码

周期性执行代码（即按固定间隔重复执行某段逻辑）是常见需求（如定时刷新数据、轮询接口、动画更新等）。实现方式主要有 setInterval、setTimeout 递归、requestAnimationFrame 等

  • 基础方法setInterval

    setInterval 是最直接的周期性执行 API，由 JS 引擎提供，核心逻辑是 “按固定间隔重复触发函数”。

    const timerId = setInterval(handler, delay, ...args);
// handler：要执行的函数（或代码字符串，不推荐）
// delay：间隔时间（毫秒），每次执行前等待的时间
// ...args：传递给 handler 的参数
// 返回值 timerId：用于后续清除定时器（通过 clearInterval(timerId)）

    执行逻辑：

    调用 setInterval 后，JS 引擎会在 delay 毫秒后首次执行 handler，之后每隔 delay 毫秒重复执行，直到 clearInterval 被调用。

    优点缺点
    语法简单，直接声明 “间隔时间”，无需手动控制下一次执行1. 执行间隔不精确：delay 是 “最小间隔”，若主线程被阻塞（如执行耗时任务），实际间隔会变长；2. 执行堆积风险：若 handler 执行时间 > delay（如 delay=1000ms，但 handler 需 1500ms），setInterval 会在 handler 未完成时触发下一次执行，导致多个 handler 实例叠加执行；3. 不适合高频动画（与浏览器刷新频率不同步，可能卡顿）

    适用于简单的周期性任务（如每秒更新一次时间、低频率轮询接口），且 handler 执行时间远小于 delay（无堆积风险）。

    例如：

    // 定义任务：打印当前时间
function printTime() {
  console.log(new Date().toLocaleTimeString());
}

// 启动定时器：每1秒执行一次
const timerId = setInterval(printTime, 1000);

// 5秒后停止（避免无限执行）
setTimeout(() => {
  clearInterval(timerId);
  console.log("定时器已停止");
}, 5000);

  • setInterval 的可靠替代 setTimeout 递归

    setTimeout 本身是 “延迟执行一次” 的 API，但通过递归调用（在 handler 内部再次调用 setTimeout），可实现周期性执行，且能避免 setInterval 的 “执行堆积” 问题。

    核心逻辑

    function repeatTask() {
  // 1. 执行任务
  console.log("执行任务...");
  
  // 2. 任务执行完后，延迟 delay 毫秒再触发下一次
  setTimeout(repeatTask, delay);
}

// 启动：首次执行
repeatTask();

    关键差异：setInterval 是 “固定间隔触发”，而 setTimeout 递归是 “上一次任务执行完后，再等 delay 毫秒触发下一次”，确保前一次任务完成后才会执行下一次，彻底避免堆积。

    优点缺点
    1. 无执行堆积风险（前一次完成才会触发下一次）；2. 实际间隔更可控（总间隔 = 任务执行时间 + delay）1. 语法稍复杂（需手动递归）；2. 间隔仍受主线程阻塞影响（但比 setInterval 更稳定）

    适用于：

    • 任务执行时间不确定（可能超过 delay），如复杂计算、网络请求（避免请求堆积）；
    • 需严格保证 “前一次完成后再执行下一次” 的场景（如批量处理数据，避免资源竞争）。

    例如：

    let timerId = null; // 用于存储定时器ID，方便清除

function repeatTask() {
  console.log("执行任务...");
  
  // 递归调用：任务完成后，1秒后执行下一次
  timerId = setTimeout(repeatTask, 1000);
}

// 启动
repeatTask();

// 5秒后停止
setTimeout(() => {
  clearTimeout(timerId); // 清除下一次待执行的定时器
  console.log("递归定时器已停止");
}, 5000);

  • 动画专用requestAnimationFrame

    requestAnimationFrame（简称 RAF）是浏览器专为UI 动画设计的 API，其执行频率与浏览器刷新频率（通常 60Hz，即约 16.7ms / 次）同步，能避免动画卡顿，是实现流畅动画的最佳选择。

    function animate() {
  // 1. 更新动画状态（如修改DOM位置、尺寸）
  element.style.left = `${x}px`;
  
  // 2. 递归调用：请求浏览器下一次重绘时执行
  requestId = requestAnimationFrame(animate);
}

// 启动动画
const requestId = requestAnimationFrame(animate);

    执行机制：浏览器每次准备重绘（约 16.7ms 一次）时，会调用 animate 函数，确保动画与屏幕刷新同步，避免 “掉帧”。

    优点缺点
    1. 与浏览器刷新频率同步，动画更流畅；2. 浏览器后台标签页时会自动降低频率（节省性能）；3. 无执行堆积风险（每次重绘仅执行一次）1. 间隔固定（约 16.7ms，无法自定义）；2. 仅用于 UI 动画，不适合非动画类周期性任务（如数据轮询）

    适用场景

    • DOM 动画（如元素移动、缩放、淡入淡出）；
    • Canvas/SVG 动画；
    • 任何需要视觉流畅更新的场景。
    const box = document.getElementById("box");
let x = 0;
let requestId = null;

// 动画函数：每次移动box 1px
function animate() {
  x += 1;
  box.style.transform = `translateX(${x}px)`;
  
  // 递归请求下一次动画
  requestId = requestAnimationFrame(animate);
  
  // 移动到300px时停止
  if (x >= 300) {
    cancelAnimationFrame(requestId);
  }
}

// 启动动画
animate();
  • Node.js 环境下的周期性执行

    Node.js 中同样支持 setInterval 和 setTimeout 递归，但需注意其事件循环机制与浏览器的差异（无 UI 渲染，更关注异步 I/O）。此外，Node.js 还提供了一些扩展方案：

    1. setInterval/setTimeout：用法与浏览器一致，但需注意：
      • Node.js 中定时器优先级低于 I/O 回调，若有大量 I/O 操作，定时器可能延迟；
      • 长时间运行的服务需确保定时器正确清理（如进程退出前）。
    1. timers 模块：Node.js 内置模块，提供更底层的定时器控制（如 setInterval 实际基于此实现），一般无需直接使用。
    1. 第三方库：如 node-schedule（支持 cron 表达式，适合定时任务，如 “每天凌晨 3 点执行”）。示例（node-schedule）：
    const schedule = require('node-schedule');

// 每小时的第0分钟执行（整点执行）
const job = schedule.scheduleJob('0 * * * *', () => {
  console.log('整点执行任务');
});

// 取消任务
// job.cancel();
  • 注意事项
    1. 必须清理定时器：所有周期性执行方法（setInterval、setTimeout 递归、RAF）若不手动停止，会无限执行，导致内存泄漏（尤其在组件化框架中）。
      • Vue 组件中：在 onBeforeUnmount 钩子中清除；
      • React 组件中：在 useEffect 的返回函数中清除。

      示例（Vue 3 中清理 setInterval）：

      import { onMounted, onBeforeUnmount } from 'vue';

export default {
  setup() {
    let timerId = null;
    
    onMounted(() => {
      timerId = setInterval(() => {
        console.log("组件运行中...");
      }, 1000);
    });
    
    onBeforeUnmount(() => {
      clearInterval(timerId); // 组件销毁前清除
    });
  }
};
    1. 避免阻塞主线程：周期性任务若执行时间过长（如复杂计算），会阻塞 JS 主线程（导致 UI 卡顿、定时器延迟）。解决方案：
      • 拆分任务为小块，用 setTimeout 分批次执行；
      • 用 Web Workers 处理计算（不阻塞主线程）。
    1. 间隔时间的选择：
      • 高频任务（如动画）：用 requestAnimationFrame（16.7ms 间隔）；
      • 低频任务（如数据轮询）：用 setInterval 或 setTimeout 递归（根据是否有堆积风险选择）；
      • 超低频任务（如每天执行）：用 cron 表达式（Node.js）或 setTimeout 计算下次执行时间（浏览器）。
  • 方法对比
    方法核心特点适用场景清除方式
    setInterval固定间隔重复执行，简单直接简单周期性任务（执行时间短）clearInterval(timerId)
    setTimeout 递归前一次完成后再执行下一次，无堆积执行时间不确定的任务（如网络请求）clearTimeout(timerId)
    requestAnimationFrame与浏览器刷新同步，适合动画UI 动画、视觉更新cancelAnimationFrame(requestId)
    Node.js node-schedule支持 cron 表达式，定时任务服务器端定时任务（如每天备份）job.cancel()
