基础面试问题

🔑 1. Vuex 基础概念

  • Q1: Vuex 是什么？它解决了什么问题？

    Vuex 是一个专为 Vue.js 应用程序开发的状态管理模式 + 库。它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化 。

    解决的核心问题是：

    • 多个组件依赖于同一状态时，多层嵌套组件传参繁琐，兄弟组件间状态传递困难。
    • 来自不同组件的行为需要变更同一状态，以往通过父子组件直接引用或事件来变更和同步状态的多份拷贝，模式脆弱，导致代码难以维护 。
  • Q2: Vuex 有哪五种核心属性？

    分别是 State、Getters、Mutations、Actions、Modules 。

  • Q3: 为什么 Vuex 的状态是响应式的？

    Vuex 的 Store 状态是响应式的，这源于 Vue.js 的响应式系统。当状态发生变化时，所有依赖此状态的 Vue 组件都会自动高效更新 。

📦 2. 核心属性详解

  • Q4: State 的作用和访问方式？
    • 作用：State 是唯一的数据源，用于存储应用的全局状态（数据），相当于组件中的 data。
    • 访问方式：
      1. 在组件中直接通过 this.$store.state.xxx访问。
      1. 使用 mapState辅助函数将 state 映射为组件的计算属性，简化书写，尤其在需要引入多个状态时非常方便 。
  • Q5: Getters 的作用和特性？
    • 作用：Getters 类似于 Vue 组件中的计算属性（computed），用于从 state 中派生出一些新的状态（例如对列表进行过滤、计数或格式化数据） 。
    • 特性：Getter 的返回值会根据它的依赖被缓存起来，且只有在它的依赖值发生了改变之后才会重新计算。Getter 方法可以接受 state作为第一个参数，也可以接受其他 getters 作为第二个参数 。
  • Q6: Mutations 的特性是什么？为什么必须是同步函数？
    • 特性：Mutations 是更改 Vuex 的 store 中状态的唯一方法。每个 mutation 都有一个字符串的 事件类型 (type) 和一个 回调函数 (handler)。Mutation 必须是同步函数 。
    • 必须同步的原因：为了确保状态的变更是可追踪的。Vue DevTools 需要为每一个 mutation 捕获状态快照。如果 mutation 支持异步操作，调试工具将无法知道状态是何时更新的，使得调试变得困难 。在组件中通过 this.$store.commit('mutationName', payload)提交 mutation。
  • Q7: Actions 与 Mutations 的核心区别是什么？
    • 职责不同：Actions 负责处理业务逻辑和异步操作（如 API 请求），而 Mutations 负责修改状态。
    • 同步/异步：Actions 可以包含任意异步操作，Mutations 必须是同步的。
    • 触发方式：Actions 通过 dispatch方法触发，Mutations 通过 commit方法提交。
    • 参数：Action 函数接收一个与 store 实例具有相同方法和属性的 context对象（包含 commit, state, getters等），而 Mutation 函数第一个参数是 state。Action 中通过 context.commit来提交 mutation，从而间接修改 state。

🧩 3. 模块化 (Modules) 与高级用法

  • Q8: 为什么要使用 Modules？

    当应用变得非常复杂时，store 对象可能会变得非常臃肿。Vuex 允许我们将 store 分割成模块（module），每个模块拥有自己的 state、mutations、actions、getters，甚至是嵌套子模块，使得代码结构更清晰，易于维护 。

  • Q9: 什么是命名空间 (namespaced)？

    默认情况下，模块内部的 action、mutation 和 getter 是注册在全局命名空间的，这可能导致不同模块的成员名称冲突。通过添加 namespaced: true可以使该模块成为带命名空间的模块，其所有内容都会自动根据模块注册的路径调整命名。在组件中访问时需要带上路径，例如 this.$store.commit('moduleA/mutationName')。

  • Q10: 页面刷新后 Vuex 数据丢失怎么办？

    因为 Vuex 的状态数据是保存在内存中的，页面刷新会重置 JavaScript 运行时环境。解决方案是状态持久化：

    1. 利用浏览器的本地存储（localStorage 或 sessionStorage），在页面刷新前（如监听 beforeunload事件）将状态保存，在应用初始化时再读取并替换回 store。
    1. 使用现成的插件，如 vuex-persistedstate，可以更方便地实现此功能 。

⚙️ 4. 实践与原理

  • Q11: Ajax 请求应该写在组件的 methods 还是 Vuex 的 actions 中？

    这取决于请求数据的用途：

    • 如果请求来的数据不会被其他组件公用，仅在当前组件内使用，那么写在组件的 methods 中即可。
    • 如果数据很大概率需要被其他组件复用，则应该将请求放在 Vuex 的 action 中，便于管理和复用 。
  • Q12: 如何在组件中使用 v-model 绑定 Vuex 的状态？

    直接使用 v-model="this.$store.state.obj.prop"是不符合 Vuex 数据流规范的。正确做法是使用计算属性的 getter 和 setter：

    computed: {
  message: {
    get() {
      return this.$store.state.message;
    },
    set(value) {
      this.$store.commit('updateMessage', value);
    }
  }
}

    然后在模板中：<input v-model="message">。

  • Q13: Vuex 的严格模式是什么？

    在创建 store 时设置 strict: true即可开启严格模式。在严格模式下，任何非 mutation 函数引起的状态变更都会抛出错误，这能保证所有的状态变更都能被调试工具跟踪到，有助于代码的规范性和可调试性。注意不要在生产环境开启，以避免性能损失 。

🔄 5. Vuex 与 Pinia

  • Q14: Vuex 和 Pinia 有什么区别？如何选择？

    Pinia 是官方推荐的下一代 Vue 状态管理库，可看作是 Vuex 5。

    • API 设计：Pinia 的 API 更简洁，移除了 Mutation，允许在 Action 中直接进行同步和异步操作来修改状态。对 TypeScript 的支持更好。
    • 模块化：Pinia 天生支持模块化，每个 store 都是独立的，无需复杂的嵌套模块。
    • 选择建议：新项目强烈推荐使用 Pinia。对于现有的 Vue 2 老项目，如果不需要升级到 Vue 3，可继续使用 Vuex；若计划升级，可逐步将 Vuex 替换为 Pinia 。

记录

  • Vuex中的state可以修改吗？
    • 可以通过this.store.state.xxx 来修改
    • 可以通过mutations 或actions 方法来修改
    • 无法通过mapState 方法来修改（mapState 没有get和set）

  • Vuex的持久化存储？

切记VUEX本身是一个存放数据的地方，并不是持久化记录数据的，但可以做持久化存储，有两种方式实现：

  • 自己写localStorage
  • 使用插件 vuex-persistedstate
npm  install --save vuex-persistedstate
