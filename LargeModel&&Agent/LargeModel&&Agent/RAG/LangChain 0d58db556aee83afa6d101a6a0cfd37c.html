LangChain调用聊天模型

聊天模型的核心任务是“展开对话”。

聊天模型就像一个懂得沟通技巧的对话专家，专门针对多轮对话场景进行过优化，核心是与用户进行交互。擅长多轮对话，自带上下文记忆，能追踪对话历史，实现连贯的交流

  • 标准流程
    1. 初始化模型
      from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model="gpt-4o",        # 指定要使用的模型名称
    temperature=0.7,       # 控制生成文本的随机性
    max_tokens=500,        # 限制单次生成的最大token数
    timeout=60,            # API请求超时时间
    max_retries=2,         # 请求失败时的重试次数
    # api_key="...",       # API密钥，建议用环境变量
    # base_url="...",      # 自定义API端点
)
    1. 构建消息列表

      LangChain通过一系列消息类来封装这些角色，主要包括：

      • SystemMessage: 定义AI的角色和回复规则，设定顶层指示。
      • HumanMessage: 代表用户的输入。
      • AIMessage: 模型生成的回复。
      • ToolMessage, FunctionMessage, ChatMessage等：用于更复杂的场景。
      from langchain_core.messages import SystemMessage, HumanMessage, AIMessage

messages = [
    SystemMessage(content="你是一个精通Python的资深工程师。"),
    HumanMessage(content="什么是LangChain？"),
]
response = model.invoke(messages)
    1. 执行调用于获取响应（.invoke(), .stream(), .batch()）

      同步调用，会等待完整结果后返回

      response = model.invoke(messages)
print(response.content)  # 输出: AIMessage 对象

      .stream()用于逐步获取生成的文本块，能提升用户体验。每个 chunk 包含新生成的一小段文本。

      for chunk in model.stream(messages):
    print(chunk.content, end="", flush=True)  # 逐字输出

      .batch()高效地处理多个独立请求的列表。

      responses = model.batch([
    ["你好，讲个笑话"],
    ["你好，讲个Python面试题"],
])
    1. 处理模型的返回

      模型返回的是一个 AIMessage 对象。它不仅包含生成的文本，还包含重要的元数据。你需要从中提取 content 和 response_metadata。

      # ... 接收 response 对象，它是一个 AIMessage
print(f"生成的文本: {response.content}")
print(f"Token使用情况: {response.response_metadata['token_usage']}")
print(f"模型名称: {response.response_metadata['model_name']}")
print(f"结束原因: {response.response_metadata['finish_reason']}")

ChatPromptTemplate

ChatPromptTemplate 是 LangChain 中专门用于构建聊天模型提示词的模板类.

它会生成一个结构化的消息列表，每条消息包含角色 role 和内容 content，更贴近聊天模型的 API 要求.

与文本语模型不同，聊天模型（如 gpt-3.5-turbo、qwen-turbo）接收一个消息数组，每条消息标记不同的发言者角色
  • 基本用法

    1. 创建ChatPromptTemplate的三种方式

    from langchain_core.prompts import ChatPromptTemplate

# 方式一：从模板字符串创建（推荐）
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个{role}助手，请用{language}回答。"),
    ("human", "{user_input}")
])

# 方式二：使用 Message 对象（更显式）
from langchain_core.messages import SystemMessage, HumanMessage

prompt = ChatPromptTemplate.from_messages([
    SystemMessage(content="你是一个{role}助手，请用{language}回答。"),
    HumanMessage(content="{user_input}")
])

# 方式三：使用模板字符串 + 角色名称（同上，语法糖）
prompt = ChatPromptTemplate.from_template(
    "system:\n你是一个{role}助手，请用{language}回答。\n\nhuman:\n{user_input}"
)

    1. 格式化生成消息列表
      messages = prompt.invoke({
    "role": "专业",
    "language": "中文",
    "user_input": "什么是 Few-shot？"

Chain链（LCEL链）

将组件串联，上一个组件的输出作为下一个组件的输入

是LangChain链（尤其是 | 管道链）的核心工作原理

实现数据的自动化流转于组件的协同工作
chain = prompt_template | model

1.核心前提：即Runnable子类对象才能入链。

2.形成的链式RunnableSerializable对象(Runnable接口子类)

3.可以通过链调用invoke或stream出发整个链条执行

  • 代码示例
    from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_community.chat_models.tongyi import ChatTongyi
from dotenv import load_dotenv
import os
from langchain_core.runnables import RunnableSerializable

chat_prompt_template = ChatPromptTemplate.from_messages(
    [
        ('system', '你是一个田园诗人'),
        MessagesPlaceholder("history"),
        ('human', '请来一首诗')
    ]
)

history_data = [
    ('human', "你来写一个唐诗"),
    ('ai', '窗前lalala'),
    ('human', '好诗，再来一个'),
    ('ai', '锄禾yiyiyi')
]

prompt_text = chat_prompt_template.invoke({
    "history": history_data,
}).to_string()

load_dotenv()

api_key = os.getenv("API_KEY")
model = ChatTongyi(model="qwen3-max", api_key=api_key)

chain: RunnableSerializable = chat_prompt_template | model
# print(type(chain)) # <class 'langchain_core.runnables.base.RunnableSequence'>

# Runnable接口，invoke执行
res = chain.invoke({"history": history_data})
print(res.content)

# Runnable接口，stream执行
for chunk in chain.stream({"history": history_data}):
    print(chunk.content, end="", flush=True)

Python中或运算的重写

只需自行实现类的__or__ 方法,即可对 | 符号的功能进行重写.

  • 实现代码
    class Test(object):
    def __init__(self, name):
        self.name = name

    def __or__(self, other):
        return MySequence(self, other)

    def __str__(self):
        return self.name


class MySequence(object):
    def __init__(self, *args):
        self.sequence = []
        for arg in args:
            self.sequence.append(arg)

    def __or__(self, other):
        self.sequence.append(other)
        return self

    def run(self):
        for i in  self.sequence:
            print(i)



if __name__ == '__main__':
    a= Test('a')
    b= Test('b')
    c= Test('c')

    d = a | b | c
    print(d, type(d))
    d.run()

Runnable接口

在 LangChain 中，Runnable 是一个核心接口（协议），它定义了 “一个可以被执行、组合、流式处理的对象” 的标准行为。

任何实现了 Runnable 的组件，都拥有统一的方法集。

  • 解决的问题

    在传统编程中，如果一个库包含多种不同的对象（如提示模板、模型、解析器、检索器），你要调用它们，得记住各自的方法名：(每个组件的调用方式不同，组合起来代码冗长且难以复用。)

    prompt_text = template.format(input=...)
response = model.generate(prompt_text)
output = parser.parse(response)

    Runnable 解决的就是这个问题：它让所有这些组件都提供 相同的方法名（invoke、stream、batch 等），并且允许它们通过管道 | 直接串联。

  • Runnable定义的方法
    方法作用输入输出
    invoke(input, config=None)同步执行，获取最终结果任意 Python 对象任意 Python 对象
    ainvoke(input, config=None)异步执行同上Awaitable[输出]
    stream(input, config=None)同步流式输出（如逐 token）同上生成器 Iterator[输出]
    astream(...)异步流式输出同上异步生成器
    batch(inputs, config=None)批量执行（多个输入）List[输入]List[输出]
    abatch(...)异步批量同上Awaitable[List[输出]]

    无论输入输出是什么，方法签名完全统一

    因此你可以把一个 Runnable 的输出直接喂给另一个 Runnable 的输入，只要类型兼容（通常是可序列化的字典或字符串）。

  • Runnable的典型实现者

    LangChain 中几乎所有核心组件都实现了 Runnable：

    组件类输入输出
    PromptTemplate / ChatPromptTemplate字典（变量名→值）PromptValue
    模型 (BaseLLM, BaseChatModel)PromptValue 或 str / List[Message]LLMResult 或 BaseMessage
    输出解析器 (BaseOutputParser)模型输出（字符串或消息）解析后的结构化数据
    检索器 (BaseRetriever)字符串查询List[Document]
    工具 (BaseTool)参数字典字符串结果

    此外，LangChain 还提供了很多 内置的 Runnable 工具 来辅助构造链：

    • RunnableLambda：把任意普通函数包装成 Runnable。
    • RunnableParallel：并行执行多个子链，并将结果合并为字典。
    • RunnableBranch：条件路由，根据输入选择执行哪个子链。
    • RunnablePassthrough：透传输入，常用于调试或旁路。

StrOutputParser字符串输出解析器

从模型的原始输出中提取出字符串文本

  • 解决的问题

    不同大模型返回的数据结构（不同的AIMessage）各不相同：

    模型类型返回对象示例包含内容
    OpenAI Chat APIAIMessage(content="你好", additional_kwargs={...})消息对象，内容在 .content 中
    通义千问 (Tongyi)AIMessage(content="你好", response_metadata={...})同上
    文本补全模型 (旧)Generation(text="你好")文本在 .text 中
    某些流式输出逐块生成 AIMessageChunk(content="你")内容也是 content 字段

    而后续大多数任务需要的只是数据结构中的一个content字符串，

    StrOutputParser负责提取这个content字段

    from langchain_core.output_parsers import StrOutputParser
from langchain_community.llms.tongyi import Tongyi
from langchain_core.prompts import ChatPromptTemplate

model = Tongyi(model="qwen-max", api_key="your-key")
prompt = ChatPromptTemplate.from_template("讲一个关于{topic}的笑话")

# 不使用解析器
chain_raw = prompt | model
result_raw = chain_raw.invoke({"topic": "程序员"})
print(type(result_raw))      # <class 'langchain_core.messages.AIMessage'>
print(result_raw.content)    # 需要手动取 content

# 使用 StrOutputParser
chain = prompt | model | StrOutputParser()
result = chain.invoke({"topic": "程序员"})
print(type(result))          # <class 'str'>
print(result)                # 直接打印字符串
  • 再LCEL链中的位置

    在典型的 LCEL 链中，结构通常是：

    from langchain_core.output_parsers import StrOutputParser
parser = StrOutputParser()

提示模板 -> 模型 -> 输出解析器(parser) -> 模型 -> ...

    如果在模型之间使用链而没有输出解析器会产生错误

自定义函数加入链

自定义函数入链 指的是通过 RunnableLambda 将一个普通的 Python 函数,转换为 LangChain 的 Runnable 对象，然后将其无缝地插入到由管道符 | 构建的执行链中。

  • 示例代码
    from langchain_core.output_parsers import StrOutputParser

    "我的邻居姓：{lastname}，刚生了{gender}，帮忙起个名字，仅告诉我名字。"
)

    "姓名：{name}, 请帮我问解析含义。"
)

# 链
chain = first_prompt | model | func | second_prompt | model | str_parser

# 运行输出
for chunk in chain.stream({"lastname": "何", "gender": "男"}):
    print(chunk, end="", flush=True)


    lambda ai_msg: {"name": ai_msg.content}
ai_msg：这是 `first_prompt | model` 的输出结果。
    - `model = ChatTongyi(...)` 是一个聊天模型，调用后返回 `AIMessage` 类型的对象（通常包含 `content` 属性，即模型生成的文本）。
    - 因为链中 `first_prompt` 将用户输入（`lastname`, `gender`）格式化为提示词字符串，然后将该字符串传给 `model`，模型返回一个 `AIMessage` 实例，其中 `content` 字段就是起好的名字（例如 `“何泽宇”`）。
返回值：`{"name": ai_msg.content}`
    - 该 lambda 将模型输出的文本提取出来，并放入一个字典中，键为 `"name"`。
    - 这个字典将作为下一个步骤 `second_prompt` 的输入。因为 `second_prompt` 模板中有变量 `{name}`，所以必须传入一个包含 `name` 键的字典。

如果想要封装历史记录，除了自行维护历史消息外，也可以借助LangChain内置的历史记录附加功能。

LangChain提供了History功能，帮助模型在有历史记忆的情况下回答。

  • 基于RunnableWithMessageHistory在原有链的基础上创建带有历史记录功能的新链（新Runnable实例）
  • 基于InMemoryChatMessageHistory为历史记录提供内存储存（临时用）

    • 输入：一个字典，必须包含 "input" 和 "chat_history" 两个键。
    • 输出：模型生成的回答文本（字符串）。

    2.增强链RunnableWithMessageHistory的作用

    base_chain,
    get_history,
    input_messages_key="input",

)

    这个包装器为 base_chain 增加了自动管理会话历史的功能。它的内部执行逻辑大致如下：

    1. 从传入的 config 中提取 session_id（例如 "user_001"）。
    1. 调用 get_history(session_id) 获取该会话对应的 InMemoryChatMessageHistory 对象。
    1. 从该历史对象中读取已有的消息列表（初始为空）。
    1. 将用户当前输入（例如 {"input": "小明有两个猫"}）与历史消息列表合并，构造一个新的字典作为 base_chain 的输入：

    1. 执行 base_chain.invoke(新字典)，获得模型回答。
    1. 将用户本轮输入和模型本轮输出作为两条新消息，追加到历史对象中。
    1. 返回模型回答。

Memory长期记忆

官方注释中给出了基于文件的历史消息存储示例代码，因此我们可以实现一个基于Json格式和本地文件的会话数据保存。

长期记忆通过 FileChatMessageHistory 类实现，其核心思路：

  • 基于文件存储会话记录，以session_id为文件名，不同session_id有不同文件存储消息
  • 继承BaseChatMessageHistory实现如下三个方法：
    • add_messages: 同步模式，添加消息
    • messages: 同步模式，获取消息
    • clear: 同步模式，清除消息

class FileChatMessageHistory(BaseChatMessageHistory):

    def __init__(self, session_id, storage_path):
        # 会话id
        self.session_id = session_id
        # 不同会话id的存储文件，所在的文件夹路径
        self.storage_path = storage_path
        # 完整路径
        self.file_path = os.path.join(self.storage_path, self.session_id)

        # 确保文件夹是存在的
        os.makedirs(os.path.dirname(self.file_path), exist_ok=True)

    def add_messages(self, messages) -> None:
        # Sequence 序列， 类似list，tuple
        all_messages = list(self.messages)
        all_messages.extend(messages)

        # 将数据同步写入到本地文件中
        # 类对象写入文件 -> 一堆二进制
        # 为了方便，可以将BaseMessage消息转为字典（借助json模块以json字符串写入文件）
        # 官方message_to_dict: 单个消息对象（BaseMessage类实例） -> 字典
        # new_messages = []
        # for message in all_messages:
        #   d = message_to_dict(message)
        #   new_messages.append(d)

        new_messages = [message_to_dict(message) for message in all_messages]
        # 将数据写入文件
        with open(self.file_path, "w", encoding="utf-8") as f:
            json.dump(new_messages, f)


    @property
    def messages(self) -> list[BaseMessage]:
        try:
            with open(self.file_path, "r", encoding="utf-8") as f:
                message_data = json.load(f) # 返回值就是: list[字典]
                return messages_from_dict(message_data)
        except FileNotFoundError:
            return []

    def clear(self) -> None:
        with open(self.file_path, "w", encoding="utf-8") as f:
            json.dump([], f)

文档加载器:CSVLoader

  • Document loaders文档加载器

    文档加载器提供了一套标准接口，用于将不同来源（CSV,PDF,JSON）的数据读取为LangChain的文档格式，这确保了无论数据来源如何，都能对其进行一致性处理。

    文档加载器（内置或自行实现）需实现BaseLoader接口。

    Class Document,是LangChain内文档的统一载体，所有文案加载器最终返回此类的实例。

    一个基础的Document类实例，基于如下代码创建：

    from langchain_core.documents import Document
document = Document(
	page_content = "Hello,world!",
	metadata={"source":"http://example.com"}
)

    如上可以看到，Document类其核心记录了:

    • page_content: 文档内容
    • metadata: 文档元数据（字典）

    不同文档加载器可能定义了不同的参数，但是其实现了统一的接口（方法）：

    • load():一次性加载全部文档
    • lazy_load():延迟流式传输文档，对大型数据集很有用，避免内存溢出

CSVLoader 是 LangChain 用来加载 CSV 文件的工具。它的核心逻辑很简单：将 CSV 文件里的每一行数据，都转换为 LangChain 框架中的一个 Document 对象

  • CSVLoader()参数：
    参数名类型作用

    source_columnOptional[str]指定来源列：指定某一列的值替换文档的 source 元数据。
    metadata_columnsSequence[str]提取元数据：从CSV中选取特定列，作为 metadata 信息存储。
    csv_argsOptional[Dict]自定义解析：一个传递给Python标准库 csv.DictReader 的参数字典，用于控制分隔符、引号等。

文档加载器:JSONLoader

JSONLoader 是 LangChain 中专为处理 JSON 和 JSON Lines 数据设计的加载器。它的强大之处在于，能通过一个关键参数 jq_schema，灵活地将复杂的、嵌套的 JSON 数据精确提取并转换为 LangChain 的 Document 对象，为后续的处理与检索做好准备。

  • JSONLoader()参数
    参数类型必填默认值一句话功能
    file_pathstr / Path✅—JSON 文件路径
    jq_schemastr❌"."jq 表达式，定位要提取的数据
    json_linesbool❌False是否为 JSON Lines 格式
    text_contentbool❌False是否将 page_content 强制转为字符串
    metadata_funcCallable❌None自定义元数据提取函数
    content_keystr❌None当提取结果为字典时，指定哪个键作为 page_content
    is_content_key_jq_parsablebool❌False是否将 content_key 视为 jq 表达式再次求值

    其中jq_schema是解析的关键，具体用法：

    jq_schema描述
    .提取整个 JSON 对象
    .[]遍历数组的每个元素
    .[].name提取数组中每个对象的 name 字段
    .data.messages[].content提取嵌套结构中的内容
    1. jq_schema : 使用 jq 语法编写的表达式，决定了从 JSON 中提取哪些数据片段。每个匹配到的片段会生成一个 Document (参数如上)
    1. json_lines : 处理 JSON Lines 格式（每行一个独立的 JSON 对象，没有外层数组包裹）。
      # 普通 JSON 数组（json_lines=False）
[
  {"id": 1},
  {"id": 2}
]

# JSON Lines（json_lines=True）
{"id": 1}
{"id": 2}
    1. text_content :

      False：page_content 保持 jq 提取结果的原始 Python 类型（字符串、数字、列表、字典、None 等）。

      True：将提取结果转换为 JSON 字符串（通过 json.dumps，对于非 JSON 可序列化对象则调用 str()）。

      需要保留数值类型用于后续数学运算 → False

      需要直接拼接字符串或传给大模型 → True（更安全）

    1. metadata_func : 允许你从原始 JSON 中提取额外字段存入 Document.metadata。
      def metadata_func(record: dict, metadata: dict) -> dict:
    """
    record:  当前 jq_schema 提取出的原始数据片段（字典或列表等）
    metadata: 包含默认元数据的字典（已有 'source' 和 'seq_num'）
    返回: 更新后的 metadata 字典
    """
      def add_author(record, metadata):
    metadata['author'] = record.get('author', 'anonymous')
    metadata['views'] = record.get('stats', {}).get('page_views')
    return metadata

loader = JSONLoader(..., metadata_func=add_author)
      metadata_func 会为每个生成的 Document 调用一次，因此要注意性能。不要在函数内部执行耗时操作（如网络请求）。
    1. content_key :

      当 jq_schema 提取的结果是一个 字典，而你只希望将字典中的某个特定键的值作为 page_content 时使用。

      loader = JSONLoader(
    file_path="data.json",
    jq_schema=".",          # 提取整个根对象 -> {"text": ..., "id": ...}
    content_key="text"      # 将 text 键的值作为 page_content
)

# 结果：doc.page_content = "hello world"
#       doc.metadata 中会保留整个原始字典? 不，默认只保留 source 和 seq_num
#       想保留其他字段要用 metadata_func
      与 metadata_func 配合：content_key 只影响 page_content；其他字段若想存入 metadata 仍需通过 metadata_func。
    1. is_content_key_jq_parsable : 一个极少使用的高级选项。当 content_key 的值本身是一个 jq 表达式，并且需要对提取出的字典再次应用该表达式时，设为 True。

文档加载器:TextLoader && 文档分割器:Text Splitter

TextLoader 是 LangChain 中最基础、最直接的文档加载器，它的主要作用就是读取 .txt、.md 这类纯文本文件，并把文件内容统一转换成 LangChain 框架可以处理的 Document 对象。

  • TextLoader()参数
    参数名类型必填/默认值核心功能
    file_pathstr / Path必填需要加载的文本文件路径，支持本地或远程文件。
    encodingstr可选，默认 None指定文件编码（如 "utf-8", "gbk"）。不指定时会尝试自动检测。
    autodetect_encodingboolFalse（默认）是否启用自动检测编码，设为 True 前需要安装 chardet。
    metadatadict可选，默认 None自定义的附加元数据，会合并到生成的 Document 的 metadata 中。
  • TextLoader()加载方式
    方法名返回值描述
    load()List[Document]一次性加载：将整个文件内容作为一个 Document 对象返回，适合小文件。
    aload()List[Document]异步加载：load() 的异步版本
    load_and_split()List[Document]在加载的同时，使用指定的 TextSplitter（如 RecursiveCharacterTextSplitter）对文本进行分块，返回多个 Document 对象，适合大文件或构建 RAG 应用
    lazy_load()Iterator[Document]惰性加载：返回一个迭代器，按需逐个生成 Document，适合处理超大文件，可有效节省内存
  • 案例
    from langchain_community.document_loaders import TextLoader

# 1. 加载文件
loader = TextLoader("example.txt", encoding="utf-8")
# 输出为 Document 列表
documents = loader.load()

# 2. 打印加载结果
print(f"Documents 数量: {len(documents)}")
print(f"内容: {documents[0].page_content}")
print(f"元数据: {documents[0].metadata}")
    • page_content: 包含文件的全部文本内容。
    • metadata: 默认包含 source (文件路径) 这一项元数据。如果需要添加更多信息，如文件来源、作者等，可以通过 metadata 参数进行自定义。

文档分割器（Text Splitter）的核心要务是：在尽可能不破坏语义的前提下，将超长文档分解成多个小“文本块”（Chunk），以适应大模型有限的上下文窗口。不合理的分割会导致语义丢失或检索不准，而高质量的分割则被视为RAG效率大幅优化的基石。

  • 核心参数
    参数名作用默认值建议
    chunk_size每个文本块的最大长度（单位取决于 length_function）4000决定“块有多大”。设置为模型上下文窗口的 1/4 到 1/2。
    chunk_overlap相邻块之间重叠的长度，用于保持上下文连贯性200决定“上下文连贯性”。通常设为 chunk_size 的 10~20%。
  • 分割器按原理分类
    1. CharacterTextSplitter （基础字符分割）

      按特定分隔符和固定长度硬性分割，它简单直接但会破坏语义，适合处理日志等无关语义的文本。

    1. RecursiveCharacterTextSplitter （递归字符分割）

      LangChain 官方推荐的通用分割器。它按优先级（如["\n\n", "\n", " ", ""]）递归分割，能最大限度保持段落、句子的完整。

    1. TokenTextSplitter （Token 数量分割）

      以模型计费的 Token 为单位进行分割，输出与模型计费对齐，避免了因字符与 Token 转换不一致导致超限。

    1. SemanticChunker （语义分割）

      基于语义相似度，将语义相近的句子归入同一块。虽然质量高但速度慢，适合对语义要求极高的场景。

    1. 结构专用分割器

      针对有结构的长文本，特定分割器可以精准解析其格式，有效保留原有逻辑：

      • MarkdownHeaderTextSplitter （Markdown 结构分割）：根据标题层级分割，适合文档。
      • PythonCodeTextSplitter / RecursiveCharacterTextSplitter.from_language （代码结构分割）：识别函数、类等语法结构，适合分析代码。
      • JSONLoader / split_json （JSON 结构分割）：提取 JSON 节点，适合结构化数据。
  • 分割器按实用场景分类
    应用场景推荐分割器推荐理由
    通用问答 RAGRecursiveCharacterTextSplitter语义保留好，鲁棒性强，适合处理书籍、文章等混合文档。
    代码分析/辅助RecursiveCharacterTextSplitter（指定语言）PythonCodeTextSplitter能识别编程语言的语法结构，不会在函数中间切断。
    Markdown 文档MarkdownHeaderTextSplitter能提取出结构化的文本块，检索时能提供明确的标题和层级上下文。
    长文本摘要/翻译TokenTextSplitter计算准确，输出与计费标准对齐，可最大限度利用上下文窗口。
    社交媒体/短文本CharacterTextSplitter结构简单，实现简单，性能开销低，满足快速分割需求。
    中文/无空格语言RecursiveCharacterTextSplitter中文等语言没有词边界，需要自定义分隔符（如句号、逗号）来避免单词被切断。
    语义精确检索SemanticChunker能最大程度保证相关语义不被分割，但会牺牲一些处理速度。
  • 案例
    from langchain_text_splitters import RecursiveCharacterTextSplitter

text = "这是一段很长的文本..." # 假设这是一篇长文章

# 1. 初始化一个分割器

    chunk_size=500,       # 每个块最大 500 字符
    chunk_overlap=50,     # 块与块之间重叠 50 字符
    length_function=len,  # 默认使用 len() 计算长度
    separators=["\n\n", "\n", "。", "，", " ", ""]   # 中文优化分隔符
)

# 2. 对文本进行分割
chunks = text_splitter.split_text(text)

# 3. (可选) 如果是从加载器得到 Document 列表
# from langchain_community.document_loaders import TextLoader
# loader = TextLoader("my_doc.txt")
# documents = loader.load()
# docs = text_splitter.split_documents(documents)

print(f"文本被分割为了 {len(chunks)} 个块")

块大小不是越小越好：在问答场景，增大块尺寸通常能提升检索命中率，但响应延迟也会增加。推荐以 400-800 token (约 300-600 汉字) 为起点调试。

重叠不能太多：重叠建议设为 chunk_size 的 10-20%。过高的重叠会显著增加索引存储和计算开销。

衡量长度用什么？：如果使用 TokenTextSplitter，块大小建议设为模型上下文窗口的 1/4 到 1/2；如果使用默认的 RecursiveCharacterTextSplitter（按字符），则需考虑中英文差异，英文可设置大一些（如 1000），中文推荐 500-800。

根据大模型微调：针对不同模型调整块大小，例如 Claude 3 的 200K 上下文可用更大块；针对嵌入模型优化时，参考其文档（如 text-embedding-3-small 训练时常用 256-512 token）。

文档加载器:PyPDFLoader

PyPDFLoader 是 LangChain 中处理可复制文本型 PDF 最直接、最基础的工具，它通过底层 pypdf 库来提取文本。

不过，它只是一个功能简单的“入门”工具，遇到扫描版 PDF 或复杂的表格时，能力会受限。

核心逻辑是：将 PDF 文件作为输入，解析其中的文本内容（非图片），并输出Document 对象。在构建 RAG（检索增强生成）系统时，它通常位于整个流程的起点。

  • PyPDFLoader()参数
    参数名类型默认值核心功能 / 说明
    file_pathstr / Path无（必填）PDF 文件路径，支持本地路径或 HTTP/HTTPS 的远程 URL。
    modestr"page"提取模式。"page"：按页加载，每页一个Document；"single"：仅返回一个合并全文的Document。
    passwordstrNone用于打开受密码保护的加密 PDF 文件。
    extract_imagesboolFalse是否尝试从 PDF 中提取图片（需额外配置解析器），常用于多模态 RAG 场景。
    headersdictNone当file_path为远程 URL 时，可以传入此参数设置 HTTP 请求头（如 User-Agent）。
    pages_delimiterstr" "mode="single"时，合并全文使用的页间分隔符。
  • PyPDFLoader()加载方式
    方法名返回值描述
    load()List[Document]一次性加载。加载全文档，根据mode参数决定按页输出还是合并全文。
    load_and_split()List[Document]加载并分割。相当于mode='page'的load()，已较少用，推荐直接使用mode参数。
    lazy_load()Iterator[Document]惰性加载。返回一个生成器，逐页加载，处理大文件时内存占用极低。

  • 案例

    1.基础按页加载

  • Documnet对象结构

    PyPDFLoader 返回的每个 Document 对象都包含两个关键属性：

    • page_content (字符串)：存储该页面的所有纯文本内容。PDF 中的图片、复杂的表格和排版信息通常会被舍弃，仅保留文字。
    • metadata (字典)：存储辅助信息，包含两项内容：
      • 'source'：PDF 文件的路径或 URL。
      • 'page'：页码（从 0 开始计数），便于追踪内容来源。
  • 一些问题
    问题原因解决方案
    提取到空白文本或乱码PDF 是由图片扫描而成的“扫描件”，本身不含文字层。改用 OCR 方案提取文字：- PyMuPDFLoader：集成 Tesseract，可进行 OCR 识别。- UnstructuredPDFLoader：强大的开源 OCR 工具。- 商用 API（Azure/AWS）：处理海量或专业文档时可考虑。
    表格或排版全乱 / 提取效率低底层 pypdf 库会丢弃格式，只按顺序提取文字。根据需求选择：- 保留表格：使用 PDFPlumberLoader，专攻表格提取。- 提升速度：使用 PyMuPDFLoader，处理速度快数倍。- 复杂/扫描文档：使用 UnstructuredPDFLoader。
    PdfReadError: file not encrypted 或 Invalid PDF structure文件来源损坏、非标准PDF或是加密文件。用 PDF 阅读器手动验证，或利用 pdf 等 Python 库修复逻辑。
    加载速度慢 / 内存不足文件过大，load() 一次性将整个 PDF 读入内存。使用 lazy_load() 方法，利用生成器按页加载，可极大降低内存占用。

基于向量检索构建提示词

向量检索：将用户问题也转换成向量，与预先存储好的知识片段向量进行相似度计算，召回 top-k 个最相关的片段。

构建提示词：用一个模板把检索到的片段和用户问题组合起来。模板通常包含三部分：

  • 角色/指令（如“你是AI助手，基于如下知识回答”）
  • 检索到的知识（{context}）
  • 用户问题（{question}）
  • 案例
    import os
from dotenv import load_dotenv
from langchain_community.document_loaders import CSVLoader
from langchain_community.embeddings import DashScopeEmbeddings
from langchain_community.vectorstores import FAISS  # 使用FAISS作为向量数据库
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# 加载环境变量
load_dotenv()
api_key = os.getenv('API_KEY')
os.environ['DASHSCOPE_API_KEY'] = api_key

# Step 1: 加载和预处理文档
loader = CSVLoader(
    file_path='./input.csv',
    encoding='utf-8',
    source_column='source',
)
documents = loader.load()

# Step 2: 切分文档（解决长文本问题）
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每块500字符
    chunk_overlap=50,    # 块间重叠50字符
)
chunks = text_splitter.split_documents(documents)

# Step 3: 向量化并存储
embeddings = DashScopeEmbeddings(
    model="text-embedding-v2"  # 使用通义嵌入模型
)
vector_store = FAISS.from_documents(chunks, embeddings)

# Step 4: 初始化通义千问大模型
llm = ChatOpenAI(
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    model="qwen-plus",  # 通义千问增强版
    api_key=api_key,
    temperature=0.7,
)

# Step 5: 构建提示词模板
prompt_template = ChatPromptTemplate.from_template("""
你是一个专业的AI助手。请基于以下提供的知识内容回答用户的问题。
- 如果知识内容足以回答问题，请结合知识进行回答。
- 如果知识不足以回答，请如实告知，不要编造信息。

【知识内容】
{context}

【用户问题】
{question}

【回答】：
""")

# Step 6: 构建RAG问答链
retriever = vector_store.as_retriever(search_kwargs={"k": 3})  # 检索3个最相关片段

def format_docs(docs):
    return "\n\n".join([doc.page_content for doc in docs])

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt_template
    | llm
    | StrOutputParser()
)

# Step 7: 执行问答测试
response = rag_chain.invoke("阿里云的通义千问模型有哪些特点？")
print(response)

RunnablePassthrough使用

如何让向量检索入链？ → 通过 RunnablePassthrough

  • 案例
    from langchain_community.chat_models import  ChatTongyi
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.vectorstores import InMemoryVectorStore
from langchain_community.embeddings import DashScopeEmbeddings
from langchain_core.output_parsers import StrOutputParser
from dotenv import load_dotenv
import os
load_dotenv()

model = ChatTongyi(model="qwen3-max", api_key=os.getenv("QWEN3_API_KEY"))
prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "以我提供的已知参考资料为主，简介和专业的回答用户问题，参考资料:{context}"),
        ("user", "用户提问:{input}")
    ]
)

vector_store = InMemoryVectorStore(embedding=DashScopeEmbeddings(model="text-embedding-v4"))

vector_store.add_texts([
    "减肥就是少吃多练",
    "在减脂期间吃东西很重要，要清淡少油控制卡路里的摄入并运动起来",
    "跑步是很好的运动哦"
])

input_text = "怎么减肥？"

# Langchain中向量存储对象，有一个方法，as_retriever，可以返回一个Runnable接口的子类实例对象
# "k", 2 最大返回2个匹配结果
retriever = vector_store.as_retriever(search_kwargs={"k": 2})

# 转换list[Document] -> str
def format_func(docs) -> str:
    if not docs:
        return "无资料"

    formatted_docs = "["
    for doc in docs:
        formatted_docs += doc.page_content
    formatted_docs += "]"
    return formatted_docs

def print_history(full_prompt):
    print("="*20, full_prompt, "="*20)
    return full_prompt

# chain = retriever | prompt | model | StrOutputParser() 有问题
chain = (
    {"input": RunnablePassthrough(), "context": retriever | format_func}
    | prompt
    | print_history
    | model
    | StrOutputParser()
)

res = chain.invoke(input_text)
# input_text会最先提供给rettiever，由于retriever在字典中，所以input_text需要先进入字典中
# 这样input_text会分流一份给RunnablePassthrough()
print(res)

"""
retriever：
    - 输入：用户提问 str
    - 输出：向量库的检索结果 list[Document]
prompt：
    - 输入：用户的提问+向量库检索结果 dict
    - 输出：完整的提示词 PromptValue
"""
  • 案例解析
    {"input": RunnablePassthrough(), "context": retriever | format_func}

    这是一个字典，它的值是两个可运行对象（Runnable），会并行执行，最终合并成一个字典输出。

    键值含义
    "input"RunnablePassthrough()将用户输入（原始问题）原封不动地传递下去
    "context"retriever | format_func将用户输入经过检索器 → 格式化函数处理后得到上下文

    输入：用户的问题字符串（例如 "怎么减肥？"）

    输出：一个字典

    {
    "input": "怎么减肥？",
    "context": "检索并格式化后的文本字符串"
}
    • RunnablePassthrough()：一个非常简单的组件，什么都不做，只是把接收到的输入直接作为输出。这里它捕获原始的 input_text。
    • retriever | format_func：
      • retriever：前面定义的 vector_store.as_retriever(search_kwargs={"k": 2})，它的输入是用户问题，输出是 list[Document]（最多2个文档）。
      • format_func：一个自定义函数，接收 list[Document]，返回一个字符串（将文档内容拼接）。

        这个管道等价于先调用 retriever，然后把结果传给 format_func。

    prompt = ChatPromptTemplate.from_messages([
    ("system", "以我提供的已知参考资料为主，简介和专业的回答用户问题，参考资料:{context}"),
    ("user", "用户提问:{input}")
])

    prompt 是一个 ChatPromptTemplate，它期望输入一个字典，包含 context 和 input 两个键，然后格式化出完整的对话消息。

    输入：上一步输出的字典（有 context 和 input）

    输出：一个 PromptValue 对象（可以看作是组装好的消息列表，即将发送给模型的内容）

    system: 以我提供的已知参考资料为主，简介和专业的回答用户问题，参考资料:跑步是很好的运动哦 在减脂期间吃东西很重要...
user: 用户提问:怎么减肥？
