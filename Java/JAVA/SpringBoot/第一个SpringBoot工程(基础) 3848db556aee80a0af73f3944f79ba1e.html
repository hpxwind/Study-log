第一个SpringBoot工程(基础)

基础流程

  • 基础流程
    1. 创建空项目工程
    1. 设置JDK
    1. 设置maven
    1. 新建一个全新的maven模块
    1. 在pom.xml文件中添加配置：
      <parent>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-parent</artifactId>
		<version>3.5.8</version>
</parent>

      这是一个启动器，启动器名称为：parent

      名称为springboot parent starter

      为什么继承它？

      使用父项目中规定的依赖版本，比较方便。

    1. 在pom.xml中引入web开发所需的启动依赖：
      <!--        做web开发引入这个web启动器-->
<!--        底层这个启动器会关联引入web场景下的开发所需要的依赖-->
<!--        关联引入：spring-webmvc，tomcat...-->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
    1. 编写Controller
      package com.Hpc.controller;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

// @RestController = @Controller + @ResponseBody
@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello World";
    }
}
    1. 编写SpringBoot核心入口程序（main方法）
      package com.Hpc;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

// springboot主入口程序
@SpringBootApplication
public class Main {
    public static void main(String[] args) {
        // 固定代码，启动容器
        SpringApplication.run(Main.class, args);
    }
}
  • 打包
    1. springboot项目打包部署非常方便，首先需要在pom.xml文件中添加打包的插件
      <build>
	<plugins>
		<plugin>
			<groupId>org.springframework.boot</groupId>
			<artifactId>spring-boot-maven-plugin</artifactId>
		</plugin>
	</plugins>
</build>
    1. 执行mvn package命令来打包，形成jar包
    1. 在.jar同目录下，配置application.properties，可以直接替换jar包中内置的

SpringBoot采用继承而非引入依赖

父亲工程的优势:

  • 统一版本管控
    父工程预定义全套 Spring 生态及第三方依赖版本，子项目引入依赖无需写版本号，避免版本冲突。
  • 内置插件默认配置
    打包、编译、测试等 Maven 插件提前配置，子项目不用重复配置插件。
  • 依赖简化引入
    直接导入各类 starter 即可自动装配配套依赖，不用手动罗列一堆子依赖。
  • 版本升级便捷
    仅修改父工程版本，整个项目所有相关依赖批量同步升级，维护成本低。
  • 规范项目构建行为
    统一编码、JDK 编译级别、打包规则，多项目环境保持一致。

直接引入的缺陷：

  • 每个依赖必须手动指定版本，极易出现版本不兼容、jar 包冲突。
  • 插件需要自行全部配置，打包、编译配置冗余繁琐。
  • starter 配套依赖需要自行梳理引入，容易漏引、多引多余包。
  • 框架升级麻烦，需要逐个修改所有依赖版本，工作量大。
  • 多项目难统一规范，不同开发人员版本、编译配置容易混乱。
因此在继承父类的时候，切记如果父类有的不加版本号(<version>)，如果父类中没有才加版本号

SpringBoot中的启动器

SpringBoot 启动器（starter）：一组预先整合好版本的依赖包集合，引入一个 starter 就能自动导入该功能所需全部相关依赖，不用手动逐个配置、管理版本，实现快速开发。

例如：spring-boot-starter-web启动器：导入后自动包含SpringMVC、Tomcat等 web 开发全套依赖，不用自己逐个添加。

启动器分官方（spring-boot-xxx-xxx-starter）提供和非官方(xxx-xxx-spring-boot-starter)

启动器不只是在springboot中使用，在任何spring项目中都可以，是一个独立的东西
也可以自己写启动器，只需要创建一个项目，引入需要的插件，依赖，再打包程jar，引入GAV坐标即可
如果版本号手动写错，或没有写，都以spring-boot-dependencies-xxx.pom文件（BOM：物料清单）

SpringBoot中如何保证版本一致

在一个项目中向上追踪父类，最终会追踪到一个pom.xml文件，内部打包方式为pom，即(<pakaging>pom</pakaging>)

里面只有<properties> <dependencyManagement> <build> 而没有<dependency> ，因此通过这个做版本控制，版本锁定。

  • 文件详细解释

    1. 核心定位

    spring-boot-dependencies 是 SpringBoot版本统一管控中心，打包类型为 pom，只做依赖版本声明，不会被打成Jar包。

    2. 关键标签 packaging>pom

    pom 打包：该工程本身不会编译生成jar/war，仅用来管理依赖版本、统一配置，是典型的Maven父级版本管理工程。

    3. 两大核心作用节点

    1. properties
      提前定义大量版本变量（如 spring.version、mybatis.version、各种中间件版本），集中管理版本号。
    1. dependencyManagement（重点）
      里面预声明Spring全家桶、第三方组件完整坐标+对应版本；
      子项目引入这些依赖时可以省略version，自动继承这里定义好的版本，全局版本统一，避免冲突。
    注意：dependencyManagement 只是声明版本，不会实际导入依赖，子项目仍要自己写 <dependency> 才会引入包。

    4. 和 spring-boot-starter-parent 的关系

    • spring-boot-starter-parent 内部就继承了 spring-boot-dependencies；
    • 方案1（继承）：项目直接继承 spring-boot-starter-parent，最常用；
    • 方案2（导入）：无法多继承时，在pom用 <dependencyManagement> 导入 spring-boot-dependencies，同样实现版本统一。

    极简总结

    这个pom是SpringBoot全局版本花名册，预先锁定所有组件版本，以此保证整个项目所有Starter、第三方依赖版本一致，解决版本冲突问题。

任何一个springboot项目，都需要做两件事：
1. 继承springboot-start-parent…
2. 引入最核心的基础启动器：spring-boot-starter（可能是包含在其他启动器中）

springboot的核心注释

  • 简单总结
    注解核心用途
    @SpringBootApplication项目启动入口，三合一注解
    @Component/@Controller/@Service/@Repository分层注册 Bean
    @Autowired/@Resource依赖注入
    @Configuration+@Bean第三方对象配置注册
    @Value/@ConfigurationProperties读取配置文件
    @RestController、@PostMapping 等接口接收请求
    @RequestBody/@PathVariable参数接收
    @Mapper/@MapperScanMyBatis 扫描 DAO
    @Transactional数据库事务
    @ConditionalXXX条件装配
    @EnableXXX开启对应模块功能
  1. @Configuration 定义配置类，替代传统XML配置，类内部可使用@Bean注册第三方对象。
  1. @Bean 写在方法上，将方法返回对象注册为 Spring Bean，默认方法名是 Bean 名称
    @Configuration
public class MyConfig {
    @Bean
    public RestTemplate restTemplate(){
        return new RestTemplate();
    }
}
  1. @SpringBootConfiguration 使用@SpringBootApplication标注的类就是一个配置类
  1. @EnableAutoConfiguration 自动配置（默认开启），根据application.properties，根据你的路径中有哪些类，底层有一系列的条件，判断这些条件。每个配置就是一个配置类，当在springboot容器中满足某个/某些条件时候，对应的自动配置类会自动配置生效。
  1. @ComponentScan 自动扫描指定包下带组件注解的类，创建对应的Bean放入SpringIOC容器

    被扫描识别的注解：@Component,@Controller,@Service,@Repository

  1. @SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan
配置类中是可以配置bean的，并且自动纳入IoC容器的管理，等同于spring.xml中
<bean id=”date” class=”java.util.Date”>
  • 组件注册一类的

    1. @Component

    通用组件注解，不属于三层架构的普通工具类使用，标记后被 Spring 扫描创建 Bean。

    2. @Controller

    控制层（MVC），用于接收前端请求，返回页面/视图；
    不能返回 JSON 数据，返回JSON需搭配@ResponseBody

    3. @RestController

    @RestController = @Controller + @ResponseBody

    常用接口注解：所有方法默认返回 JSON 字符串，不再跳转页面。

    4. @Service

    业务层注解，写业务逻辑类，事务一般加在该层。

    5. @Repository

    持久层注解，标记 DAO/Mapper 接口，解决数据库注入爆红问题；
    MyBatis 中更多使用@Mapper替代。

  • 依赖注入注解

    1. @Autowired

    自动按类型注入 Bean，Spring 原生注解

    • 可写在成员变量、set 方法、构造方法
    • 默认必须找到对应 Bean，找不到报错；
    • @Autowired(required = false)：找不到不报错

    2. @Qualifier

    配合@Autowired使用，按名称精准匹配，解决同类型多个 Bean 冲突

    @Autowired
@Qualifier("userServiceImpl")
private UserService userService;

    3. @Resource

    JSR 规范注解，默认

    先按名称匹配，再按类型匹配,等同于@Autowired+@Qualifier，部分场景兼容性更好。

    4. @Value

    读取 application.yml/application.properties 配置文件单个值user.name=张三

    @Value("${user.name}")
private String username;
  • 配置文件批量绑定注解

    1.@ConfigurationProperties

    批量将配置文件一组属性绑定到实体类，前缀匹配

    student:
  id: 1
  name: 李四
  age: 20

    实体类：

    @Component
@ConfigurationProperties(prefix = "student")
public class Student {
    private Integer id;
    private String name;
    private Integer age;
    // get/set
}

    必须搭配@Component让类被扫描；IDEA可安装SpringBoot配置提示插件。

    2.@EnableConfigurationProperties

    开启指定类的@ConfigurationProperties绑定，不用在实体加@Component。

  • web开发常用

    1. @RequestMapping

    通用请求映射，可标注类 + 方法，指定请求路径、请求方式

    @RequestMapping("/user")
public class UserController {
    @RequestMapping("/list")
    public List<User> list(){}
}

    属性：value/path路径、method指定 GET/POST 等

    派生注解（简化写法）

    • @GetMapping = @RequestMapping(method = GET)
    • @PostMapping = POST
    • @PutMapping = PUT
    • @DeleteMapping = DELETE

    2. @RequestParam

    接收普通请求参数（?name=xxx）

    public String test(@RequestParam String name)

    3. @PathVariable

    接收路径变量 /user/{id}

    @GetMapping("/{id}")
public User getById(@PathVariable Integer id)

    4. @RequestBody

    接收前端JSON 请求体，POST 常用，自动 JSON 转对象

    @PostMapping
public void add(@RequestBody User user)

    5. @RequestHeader / @CookieValue

    获取请求头、Cookie值。

  • MyBatis/持久层相关注解

    1. @Mapper

    标记 Mapper 接口，Spring自动生成代理对象，无需写@Repository，无需 XML 扫描配置。

    2. @MapperScan（启动类常用）

    批量扫描指定包下所有 Mapper 接口，不用每个接口加@Mapper

    @SpringBootApplication
@MapperScan("com.xxx.mapper")
public class App{}

springboot中单元测

单元测试：针对最小代码单元（一个方法、一个函数）单独写代码验证逻辑是否正确Java/SpringBoot 里最小单元一般就是某个 Service 方法、工具类方法。

  • 不用单元测试：人肉手动试，慢、容易漏、改代码容易改错旧功能
  • 用单元测试：代码自动校验小方法，跑得快、防改错、便于维护迭代
  • 使用流程

    1、引入依赖（Maven）

    <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>

    2、测试类注解

    // JUnit5（默认新版）
@SpringBootTest
class TestDemo {

    @Autowired
    private UserService userService;

    @Test
    void testFunc() {
        // 调用业务方法断言
        System.out.println(userService.getById(1));
    }
}

    注解说明

    • @SpringBootTest：加载完整Spring上下文，自动扫描Bean，可@Autowired注入
    • @Test：标记测试方法

    3、只测Controller（轻量，不用完整容器）

    @SpringBootTest
@AutoConfigureMockMvc
class ControllerTest {
    @Autowired
    private MockMvc mockMvc;

    @Test
    void testApi() throws Exception {
        mockMvc.perform(get("/user/1"))
                .andExpect(status().isOk());
    }
}
