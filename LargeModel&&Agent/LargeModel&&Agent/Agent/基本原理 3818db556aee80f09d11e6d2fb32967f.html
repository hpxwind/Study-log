基本原理

一、基础定义

Agent 的标准双阶段循环：规划 Plan → 执行 Execute → 观测反馈 → 重新规划，是 ReAct、AutoGPT、BabyAGI、LangGraph、OpenAI Agent 通用核心架构。

  1. Plan 规划
    大模型作为思考者，基于目标、历史上下文、工具能力，拆解复杂任务，生成结构化行动方案（步骤、工具调用顺序、参数），不真正调用外部工具，只输出思考与行动计划。
  1. Execute 执行
    程序运行时层，读取 Plan 输出的行动指令，调用真实工具/接口/代码/数据库，完成实际操作，返回工具原始结果给 Agent。

二、核心原理

1. Plan 规划层原理（LLM 驱动推理）

核心作用：解决“大模型不能一步完成复杂任务”

LLM 上下文有限、无法直接操作外部世界，复杂目标（数据分析、爬虫、写代码、多步骤调研）必须拆分。

规划底层逻辑分三类主流范式

（1）链式规划 ReAct Plan（最简单）

Prompt 约束模型输出固定格式：Thought(思考) → Action(行动) → Action Input(参数)

  • 输入：用户目标 + 已完成步骤结果 + 可用工具列表
  • LLM 推理：分析当前还差什么信息、下一步该调用哪个工具
  • 输出：单步行动计划（一次只规划一步）
  • 特点：每执行完一步再重新规划，适合动态、不确定场景

示例 Plan 输出：

Thought: 需要先查询2025年GDP数据才能对比
Action: search
Action Input: 2025中国GDP总量

（2）整体规划 Plan-and-Solve / BabyAGI 全局规划

一次性生成完整多步骤清单，先把全流程方案全部列出来，再依次执行。
流程：

  1. LLM 接收总目标，输出完整分步列表：步骤1、步骤2、步骤3…
  1. Execute 按顺序逐条执行，中途失败则重新 Plan 修正流程
    适用：流程固定、任务可提前预知（文档整理、批量文件处理）

（3）分层规划 Hierarchical Plan（复杂长任务）

  • 高层 Plan：拆解子任务（宏观阶段）
  • 底层 Plan：每个子任务再拆解工具调用步骤
    适合超长任务：项目开发、论文撰写、多轮复杂调研（AutoGPT 核心机制）

Plan 层关键约束原理

  • 工具约束：LLM 必须知晓所有可用工具名称、入参、返回格式，否则生成无效 Action
  • 上下文记忆：传入历史执行结果，避免重复调用工具
  • 反思修正：若上一步 Execute 返回失败/无效数据，Plan 会调整下一步行动

2. Execute 执行层原理（程序调度层，无LLM推理）

Execute 是代码运行时调度模块，不做思考，只负责落地 Plan 的指令，属于“工具调度器”。
完整执行链路：

  1. 解析 Plan 输出的结构化 Action（工具名、参数）
  1. 参数校验：类型、必填项、权限校验，过滤非法调用
  1. 路由分发：匹配对应工具函数
    • 内置工具：代码解释器、计算器、文件读写
    • 外部API：搜索引擎、数据库、第三方接口
    • 子Agent：调用其他子智能体
  1. 真实运行工具，捕获返回结果、异常报错（超时、接口404、权限不足）
  1. 标准化封装观测 Observation，回传给 Plan 层作为下一轮输入

Execute 核心能力

  1. 隔离 LLM 与外部环境：LLM 无法直接访问网络/文件，所有IO操作由 Execute 代理，保证安全
  1. 异常容错：工具调用失败时捕获错误，把异常信息丢给 Plan，让大模型重新规划补救方案
  1. 并行/串行调度：支持顺序执行、批量并行调用工具，提升效率
  1. 权限管控：限制文件读写、网络访问范围，防止Agent危险操作

三、Plan + Execute 完整闭环工作流（标准循环）

1. 用户输入总目标
↓
2. Plan（LLM）：分析目标，生成行动计划（单步/全流程）
↓
3. Execute（程序调度）：解析行动，调用真实工具，拿到原始数据
↓
4. Observation 观测：格式化工具返回结果
↓
5. 送入Plan层上下文：LLM结合已有结果重新思考
    - 任务完成：终止循环，整理答案输出
    - 信息不足/执行失败：重新生成新Plan，重复循环

极简例子：查询北京今日气温

  1. 用户目标：查北京今天温度
  1. Plan阶段（LLM思考）：需要调用天气查询工具，城市参数北京
    Plan输出：Action=weather_query, city="北京"
  1. Execute阶段：
    • 解析工具名 weather_query，参数北京
    • 调用天气API接口，获取原始数据 {"temp":28,"weather":"晴"}
  1. Observation 回传：当前北京气温28度，晴天
  1. Plan再次判断：信息足够，无需继续执行，整理结果回复用户

四、Plan 与 Execute 核心区别对比

维度Plan 规划Execute 执行
运行主体大模型 LLM后端代码/调度器
核心能力逻辑推理、任务拆解、决策IO调用、工具调度、异常处理
是否操作外部资源否，纯思考文本输出是，真实访问接口、文件、数据库
输出内容结构化行动指令（Thought/Action）工具原始返回数据、报错信息
失败处理重新推理、调整方案捕获异常、记录错误，交给Plan修复
依赖Prompt、记忆、工具描述函数注册、网络、文件系统权限

五、主流框架中 Plan & Execute 实现

  1. LangChain
    • Plan：PlanAgent、ReActAgent 的 LLMChain 负责思考生成Action
    • Execute：ToolExecutor / AgentExecutor 执行工具调用，封装循环
  1. LangGraph
    节点拆分：plan_node（LLM推理节点） + execute_node（工具调用节点），图状态流转实现循环
  1. AutoGPT / BabyAGI
    分层规划模块输出任务列表，执行器循环遍历任务逐条运行
  1. OpenAI Agent SDK
    model 输出 tool_calls（Plan），runtime 执行 tool 函数（Execute）

六、关键设计要点

  1. Plan 输出必须结构化（JSON/固定标记格式），否则 Execute 无法解析
  1. 分离思考与执行解耦：更换LLM只改Plan层，新增工具只改Execute层
  1. 长任务分层规划：避免一次性规划过长，超出LLM上下文窗口
  1. Execute 增加缓存：重复工具查询直接返回缓存，减少LLM+API消耗
  1. 规划反思机制：每次执行后让Plan复盘上一步结果，优化后续行动路径
