段（Segment）和节（Section）

用于组织数据和代码的核心结构，但其作用层级和用途不同。

段主要面向操作系统加载器，用于内存分配管理

节主要面向编译器和链接器，用于代码和数据的逻辑划分

  • 概念相关区别
    维度段（Segment）节（Section）
    定义层面面向操作系统加载器，描述二进制文件在内存中的布局（如何加载到内存）。面向编译器 / 链接器，描述二进制文件的逻辑组成（代码、数据、符号表等）。
    粒度较粗，一个段可包含多个节（将性质相似的节合并）。较细，按功能划分（如代码节、数据节、调试信息节等）。
    核心作用告诉操作系统：加载后哪些区域可执行、哪些可读写、哪些只读。告诉编译器 / 链接器：代码 / 数据的存储位置、符号信息、重定位信息等。
    常见格式在 ELF（Linux）、PE（Windows）等格式中均存在，结构依赖文件格式。同段，依赖文件格式（如 ELF 的 .text、.data，PE 的 .text、.rdata）。
  • 节（Section）逻辑划分的最小单元

    节是按照功能划分的逻辑单元，由编译器在编译时生成，链接器在链接时合并或处理。不同类型的节存储不同数据，常见节如下（以 ELF 格式为例）：

    • 代码相关节

      .text 存放可执行代码（如函数体），加载到内存后通常为只读且可执行（防止意外修改）。

      例：main 函数、库函数的机器码都存于此。

      .plt 过程链接表，用于动态链接时的函数地址跳转（如调用共享库中的函数时，先通过 .plt 找到实际地址）.

      .init / .fini 存放初始化代码和终止代码（程序启动时自动执行初始化，退出时执行清理）

    • 数据相关节

      .data 存放已初始化的全局变量和静态变量（有初始值），加载后为可读可写

      例：int a = 10;（全局变量）存于此。

      .bss 存放未初始化的全局变量和静态变量（初始值为 0 或未定义），仅记录大小，不占用文件磁盘空间（加载到内存时分配零初始化的空间）

      例：int b;（全局变量）、static int c; 存于此

      .rodata 存放只读数据（如字符串常量、const 变量），加载后为只读（防止修改）

      例：printf("hello") 中的 "hello" 字符串存于此\

      .data.rel / .data.rel.local 存放需要重定位的数据（如全局变量的地址引用，链接时需修正地址）

    • 符号与调试相关节

      .symtab / .dynsym 符号表，存放变量名、函数名与地址的映射（.symtab 包含所有符号，.dynsym 仅包含动态链接相关符号）。
      .strtab / .dynstr 字符串表，存储符号表中用到的字符串（如函数名、变量名），.symtab 仅记录字符串在 .strtab 中的偏移量（节省空间）
      .debug / .line 存放调试信息（如源代码行号与机器码地址的映射），供调试器（如 gdb）使用， release 版本通常会剥离这些节以减小文件体积

      .rel.text / .rela.text 重定位表，记录 .text 节中需要重定位的位置（如函数调用的地址在链接时需修正为实际地址）。

    • 其他节

      .note 存放注释信息（如编译器版本、操作系统信息）。

      .shstrtab 节头字符串表，存储所有节的名称（如 .text、.data 的名字）

  • 段（Segment）内存加载的单元

    段是操作系统加载二进制文件到内存时的管理单元，由链接器根据节的属性（如读写权限、是否可执行）合并生成。

    一个段可包含多个节

    核心是通过 “段头表” 告诉操作系统：加载后内存区域的权限和大小

    • LOAD段：可加载到内存的段（最核心）

      ELF 中最多的段类型，包含需要加载到内存的代码和数据，通过权限标志区分：

      • 代码段（通常权限 R-X）：包含 .text、.plt 等可执行且只读的节，加载到内存后可执行指令。
      • 数据段（通常权限 RW-）：包含 .data、.bss 等可读可写的节，加载后用于存储变量。
      • 只读数据段（通常权限 R--）：包含 .rodata 等只读数据，加载后不可修改。
      注：.bss 节通常合并到数据段中，加载时由操作系统分配零初始化内存。
    • DYNAMIC段：动态链接信息

      包含动态链接所需的元数据（如依赖的共享库路径、动态符号表位置），供动态链接器（如 ld-linux.so）使用。

    • NOTE段：辅助信息

      包含程序的注释信息（如操作系统、编译器信息），与运行无关，仅用于标识

    • GNU_STACK 段：栈权限信息

      指定程序栈的权限（如是否可执行，现代系统通常设为 RW- 防止栈溢出攻击）

  • 节与段的关系

    包含关系：段是节的 “容器”，一个段可包含多个节（按权限或功能合并）。例：LOAD 段（权限 R-X）可能包含 .text、.plt 节；LOAD 段（权限 RW-）可能包含 .data、.bss 节。

    映射关系：ELF 文件通过 “节头表”（Section Header Table）描述节，通过 “段头表”（Program Header Table）描述段，段头表中会记录该段包含哪些节。

    ELF 文件
├─ 段头表（描述段）
│  ├─ LOAD 段（R-X） → 包含 .text、.plt 节
│  ├─ LOAD 段（RW-） → 包含 .data、.bss 节
│  └─ DYNAMIC 段 → 包含 .dynamic 节
└─ 节头表（描述节）
   ├─ .text 节（代码）
   ├─ .data 节（已初始化数据）
   ├─ .bss 节（未初始化数据）
   └─ ...（其他节）
  • 不同文件格式的差异

    ELF（Linux/Unix）：明确区分段（Program Header）和节（Section Header），段用于加载，节用于链接。

    PE（Windows）：没有 “节” 的明确定义，而是用 “节（Section）” 同时承担逻辑划分和内存加载的角色（类似 ELF 的 “节”，但加载时直接作为内存块），通过节的属性（如 IMAGE_SCN_MEM_EXECUTE）控制权限。

    Mach-O（macOS/iOS）：使用 “段（Segment）” 和 “节（Section）”，段是顶级容器，节嵌套在段中（如 __TEXT 段包含 __text、__cstring 节），与 ELF 逻辑类似。
