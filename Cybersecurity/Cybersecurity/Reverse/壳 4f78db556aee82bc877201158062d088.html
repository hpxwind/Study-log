壳

  • 基础了解

    加壳的目的：

    1，保护软件程序代码不被逆向（加密壳，病毒比较喜欢用）

    2，减小程序文件体积（压缩壳）

    一般加壳软件会将原始程序数据（代码，导入表，资源）加密，嵌入到外壳程序内部。当程序运行时，外壳程序在内存中对已经加密的数据解密，并引导执行原始程序。原始程序的入口点称 OEP。

    脱壳方法：

    1，自动脱壳机（不推荐）

    2，手动脱壳（万能，过程繁琐）

    最直接的脱壳：

    找到 OEP （原始程序入口点：未加壳的程序的真正入口点，手动脱壳的目标）

    对应的EP（程序的入口点）

    （CTF中大部分加壳强度并不是很大，我们可以直接dump内存中的可执行文件，即可执行文件静态分析，若要进行调试，则最好找到 OEP 或者 main 函数地址后设置硬件断点进行调试）

    # 手动脱壳：
1. 打开 x64dbg，加载 `Dll1.dll`（或附加到加载该 DLL 的进程）。  
2. 在反汇编窗口中，搜索 UPX 特征码（如 `55 48 8B 0D ? ? ? ?`）。  
3. 找到 UPX 解压循环的关键指令（如 `jmp`、`call`），设置断点。  
4. 运行程序，等待断点触发，观察寄存器变化，定位 OEP（原始入口点）。  
5. 右键点击内存区域 → `Memory` → `Dump`，将 DLL 内存数据保存为 `dumped.dll`。  
6. 使用 ImportREC64 修复导入表：  
   - 选择目标进程，点击 `AutoSearch` 扫描 IAT。  
   - 点击 `Get Imports` → `Fix Dump`，生成修复后的 DLL。

    在入口点设置硬件断点

    普通软件断点会修改原始代码数据为断点指令的数据，在程序启动阶段，OEP 处的数据为加密后的数据，此时修改成断点指令，壳会对断点指令数据修改，变成其他非法指令，因此必须使用不会修改代码数据的硬件断点。

    硬件断点设置方法：

    1，先设置成普通断点

    2，再右键断点处选择 【Edit Breakpoint】

    3，选择 Hardware

  • 断点
    • 普通断点

      在一般调试器下，在原来的代码的基础上，将原来的代码改为0xcc（也有可能是其他值，只要能正常触发异常就行，这样调试器就可以让程序断下来。）

      只要关闭值为0xcc的进程，就可以恢复。

    • 内存断点
      BOOL VirtualProtect(  
  LPVOID lpAddress,  // lpAddress，要改变属性的内存起始地址。
  DWORD dwSize,  // dwSize，要改变属性的内存区域大小。
  DWORD flNewProtect,  // flNewProtect，内存新的属性类型，设置为PAGE_EXECUTE_READWRITE（0x40）时该内存页为可读可写可执行。
  PDWORD lpflOldProtect  // pflOldProtect，内存原始属性类型保存地址。
);

      可以改变某个分页的属性，读，写，执行。

      如：访问断点，可以将某个分页的属性设置为不可读，那么程序一旦去读取这块内存，就会引发异常，从而被调试器断下来。

      但有一个缺点：就是设定断点是有范围的，，如：我们需要访问0x10000时断下来，但是由于下断点是下在一个分页上的，就会导致问题，如果分页是0x1000，那么访问0x10000~0x11000这个范围都会被断下来。

    • 硬件断点

      cpu有 特定的调试寄存器，其中使用4个调试寄存器（DR0，DR1，DR2，DR3）来设定地址，以及DR7设定状态。

      硬件断点也有访问，写入，执行三种，相比于内存断点，硬件断点可以更精确的下断点，可以在某一个地址下，而不是一整块内存。

      缺点显而易见，就是硬件限制只有四个调试寄存器可以使用，所以硬件断点最多只能有四个。

      1、访问断点：对内存地址进行读操作，或写操作时。比如追踪程序对特定内存区域的使用情况时使用，当你知道程序在某处读取或修改某个局部变量的值时，就可以在此设置访问断点，方便观察程序的，寄存器的，其他变量的状态。

      2、执行断点：只有当程序执行到设置断点的内存地址处的指令时，才中断。通常用于分析程序的关键代码部分。比如在调试一个复杂的函数时，可以在函数的入口处设置执行断点，这样在调试的时候，会从内部的第一条指令处中断，这样就可以一步一步观察每条指令的效果。

      3、写入断点：仅当程序对设置断点内存进行写操作时触发。在关注程序在那些地方修改了特定的内存区域时使用（缓冲区，一个数据结构中的特定字段），例如当你怀疑程序存在缓冲区溢出漏洞时，可能在缓冲区的内存地址处设置写入断点，一旦代码往该缓冲区写入超出范围的数据，调试器选择断点类型和设置断点位置。

      —>对应UPX之类的这种压缩壳有效，不破坏原来的代码才行。

      基本操作就是不要往回跳，一直往下走，遇到不明白的Call也不需要进去，只需要观察这个Call有无实际作用，没有就不进去。

    • 2、API定位法：

      需要经验，需要对于厂商的编译器有所了解，知道OEP附近调用了什么API，

      例如：VC6写的程序，只需要定位 GetVersion 这个API即可，然后翻一下上下文就行。

    • 3、ESP定律法：

      一般压缩壳都会有pushad，

      开始运行的时候，一些壳要先保证原来的栈环境不被破坏，于是我们可以找有没有pushad这个指令，有的话我们直接找popad即可，

      这样在附近有一个大跳，一般就是跳到OEP了（如果没有popad，那么可以根据pushad完之后，在当前esp下断点）

      为什么在pushad完下断点？

      因为保存的栈结构在运行壳程序后，是不会再动用的，所以一旦程序去访问了pushad进来的栈帧结构，就可以认为要恢复栈结构了，就是要准备跳到代码段了。

      为什么要下硬件断点？

      首先要明白什么时候断下，是在访问这个栈帧的时候把它断下，所以我们只能下内存访问断点，或硬件断点，只有这两类的断点是可以下访问断点的。

    • 4、暂停定位法：

      OEP不好找，暂停之后慢慢栈回溯

  • 手动脱ASPack壳
    • ASPack壳

      通过代码优化何数据压缩减小EXE，DLL文件体积。具有反调试，反汇编，增加逆向工程难度。

      1. 压缩机制
        • 代码重组：将程序代码重新排列，移除冗余数据。
        • 数据压缩：使用自定义算法压缩资源、字符串等数据。
        • 运行时解压：程序启动时，压缩数据在内存中动态解压执行。
      1. 保护机制
        • 入口点混淆：修改程序入口点（OEP, Original Entry Point），指向解压代码。
        • 反调试检测：包含简单的反调试逻辑（如检查 IsDebuggerPresent）。
        • 控制流混淆：通过跳转和分支指令扰乱静态分析工具的控制流图。
      1. 文件特征
        • PE 头中的 SizeOfImage 显著减小。
        • 可能包含 .aspack 或 .adata 节（非标准节名）。
      1. 运行时特征
        • 程序启动时内存占用会突然增加（解压过程）。
        • 入口点附近有密集的内存操作指令（MOV、XOR、CALL）。
      1. 静态分析特征
        • IDA Pro 等工具难以正确识别函数边界。
        • 字符串和导入表被加密，动态解析。
    • ESP定律：

      1、在程序入口点下断点

      2、执行到ESP寄存器值稳定（不再变化）

      3、在ESP地址下硬件断点（右键ESP→“Hardware breakpoint on write”）

      4、继续执行直到触发断点，此时接近OEP
