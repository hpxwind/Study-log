建造者模式与@Builder注解

建造者模式（Builder Pattern）属于创造型设计模式，GoF23种设计模式之一

用于解决对象创建时参数过多的问题，它可以让对象的构造过程逐步完成，而不是一次性提供所有参数。

建造模式的主要目的是让对象的创建过程更加清晰，灵活和可控。

简而言之，建造模式属于：

  1. 简化构造过程：通过逐步构造对象，避免构造函数参数过多
  1. 提高可读性与可维护性：让构造过程更加清晰和有序
  1. 增强灵活性：允许按需配置对象的不同部分
  • 手写构造结构code
    // 目标实体
class User {
    private Long id;
    private String name;
    private Integer age;
    private String email;

    // 私有构造，只能由Builder构建
    private User(Builder builder) {
        this.id = builder.id;
        this.name = builder.name;
        this.age = builder.age;
        this.email = builder.email;
    }

    // 静态内部建造者
    public static class Builder {
        private Long id;
        private String name;
        private Integer age;
        private String email;

        public Builder id(Long id) {
            this.id = id;
            return this; // 链式返回自身
        }
        public Builder name(String name) {
            this.name = name;
            return this;
        }
        public Builder age(Integer age) {
            this.age = age;
            return this;
        }
        public Builder email(String email) {
            this.email = email;
            return this;
        }
        // 最终构建对象
        public User build() {
            return new User(this);
        }
    }

    // getter
    public Long getId() { return id; }
    public String getName() { return name; }
    public Integer getAge() { return age; }
    public String getEmail() { return email; }
}
  • 使用方式
    User user = new User.Builder()
        .id(1L)
        .name("张三")
        .age(20)
        .email("zs@qq.com")
        .build();
    • 链式调用，代码清晰易读
    • 灵活选择性赋值，不必传所有参数
    • 对象创建过程可控，可在build()中做参数校验
    • 构造逻辑与实体解耦

@Builder即自动构建建造者模式

  • 引入xml依赖
    <dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.30</version>
    <scope>provided</scope>
</dependency>
  • 简单使用
    import lombok.Builder;
import lombok.Getter;

@Builder
@Getter
public class User {
    private Long id;
    private String name;
    private Integer age;
    private String email;
}

    调用：

    User user = User.builder()
        .id(1L)
        .name("张三")
        .age(20)
        .email("zs@qq.com")
        .build();
  • @Builder原理

    编译期自动生成：

    1. 静态内部 UserBuilder 类
    1. 每个字段对应同名赋值方法，支持链式返回
    1. build() 方法创建实例
    1. 全参构造（默认私有）

    等价于上面手写完整 Builder 代码，省去重复模板。

  • 常见问题

    1. 无参构造丢失问题

    @Builder 默认只生成全参私有构造，框架（Spring、MyBatis、Jackson）反射需要无参构造会报错。
    解决：搭配 @NoArgsConstructor + @AllArgsConstructor

    @Builder
@NoArgsConstructor
@AllArgsConstructor
@Getter
public class User {
    private Long id;
    private String name;
}

    ⚠️ 此时编译会报错，需要用 @Builder.Default 或手动兼容，推荐方案：

    @Getter
@NoArgsConstructor
public class User {
    private Long id;
    private String name;

    @Builder
    public User(Long id, String name) {
        this.id = id;
        this.name = name;
    }
}

    2. 默认值失效问题

    直接写默认值，Builder 构建时会被覆盖：

    // 错误
@Builder
private Integer status = 1;

    使用 @Builder.Default 声明默认值：

    @Builder
private @Builder.Default Integer status = 1;

    3. 静态字段、final字段

    • static 字段不会生成 builder 赋值方法
    • final 字段可以正常构建

    4. 父类字段无法构建

    子类 @Builder 默认不能给父类属性赋值，解决方案：

    1. 父类也加 @Builder
    1. 使用 @SuperBuilder（Lombok 1.18+）
    @SuperBuilder
@Getter
class Parent {
    private String parentName;
}

@SuperBuilder
@Getter
class Son extends Parent {
    private String sonName;
}
// 使用
Son.builder().parentName("父").sonName("子").build();

    5. 拷贝对象（toBuilder = true）

    @Builder(toBuilder = true)
@Getter
public class User {
    private Long id;
    private String name;
}
// 基于原有对象复制，修改个别字段
User old = User.builder().id(1L).name("张三").build();
User newUser = old.toBuilder().name("李四").build();

配合@Builder的@Singular注释

@Singular专门针对集合，让其构建时候像建造模式

@Singular必须配合 @Builder 使用，专门优化集合类型字段的链式构建：

  • 不加@Singular时，Builder 只能一次性传入整个集合；
  • 加上后可以逐个追加元素，写法更优雅，同时自动生成批量添加、清空方法。
  • 代码定义
    import lombok.Builder;
import lombok.Singular;
import lombok.Getter;
import java.util.List;
import java.util.Map;
import java.util.Set;

@Builder
@Getter
public class Student {
    private Long id;
    private String name;

    // List 集合
    @Singular
    private List<String> hobbies;
    // Set 集合
    @Singular
    private Set<String> tags;
    // Map 集合
    @Singular
    private Map<String, Integer> scoreMap;
}

    字段 private List<String> hobbies; 加 @Singular 后，Builder 内部自动生成 3 个方法：

    1. 单数添加方法：hobby(String item) 追加单个元素，链式返回 Builder
    1. 批量添加方法：hobbies(Collection<String>) 追加一整个集合
    1. 清空方法：clearHobbies() 清空已添加所有元素

    ⚠️ 不再生成 hobbies(List) 直接覆盖赋值的 setter

  • 构建调用
    Student student = Student.builder()
        .id(1L)
        .name("小明")
        // 逐个添加单个元素（单数方法 hobby()）
        .hobby("游泳")
        .hobby("打球")
        // 批量追加整个集合（复数方法 hobbies()）
        .hobbies(List.of("看书", "画画"))

        .tag("好学生")
        .tag("班干部")

        // Map 键值对逐个添加
        .scoreMap("语文", 90)
        .scoreMap("数学", 95)
        .build();

@Slf4j注解生成日志常量

@Slf4j直接使用log打印日志

  • 引入依赖
    <!-- lombok -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.30</version>
    <scope>provided</scope>
</dependency>

<!-- slf4j 门面 -->
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-api</artifactId>
    <version>1.7.36</version>
</dependency>

<!-- 实现（logback/log4j2 二选一，项目一般已有） -->
  • 使用
    import lombok.extern.slf4j.Slf4j;

@Slf4j
public class UserService {

    public void test() {
        log.info("普通信息日志");
        log.error("异常日志", new RuntimeException("出错了"));
        log.debug("调试日志");
        log.warn("警告日志");
    }
}

Lombok 在编译期自动插入静态常量 logger，反编译等价代码：

  • code
    import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    // 自动生成：private static final Logger log
    private static final Logger log = LoggerFactory.getLogger(UserService.class);

    public void test() {
        log.info("普通信息日志");
        log.error("异常日志", new RuntimeException("出错了"));
        log.debug("调试日志");
        log.warn("警告日志");
    }
}

    关键特征

    1. 修饰符：private static final
    1. 变量名默认：log
    1. 构造：LoggerFactory.getLogger(当前类.class)
    1. 属于类静态常量，全局唯一，不会重复创建 Logger
  • 同类日志注解对比（都是生成日志常量，只是实例名 / 工厂不同）
    注解生成代码适用日志框架
    @Slf4jprivate static final Logger log = LoggerFactory.getLogger(当前类.class);SLF4J + Logback/Log4j2
    @Log4jprivate static final Logger log = Logger.getLogger(当前类.class);Log4j1
    @Log4j2private static final Logger log = LogManager.getLogger(当前类.class);Log4j2
    @CommonsLogprivate static final Log log = LogFactory.getLog(当前类.class);Apache Commons Logging
    @FloggerGoogle Flogger 日志Google 日志

自定义日志变量名（Lombok 配置修改 log 变量名）

默认变量是 log，如果项目规范要求必须叫 logger，有两种方式：

  • lombok.config 全局配置

    在项目根目录新建 lombok.config

    # 修改 @Slf4j 生成的变量名
lombok.log.fieldName = logger

    编译后代码变为：

    private static final Logger logger = LoggerFactory.getLogger(UserService.class);
  • 手动自己写 Logger
    public class UserService {
    private static final Logger logger = LoggerFactory.getLogger(UserService.class);
}
  • 注意事项

    1. 不能在静态代码块之前使用 log

    @Slf4j
public class Demo {
    // 编译报错：变量未定义
    static {
        log.info("静态块");
    }
}

    原因：Lombok 生成的 log 常量在类成员变量区，静态代码块执行顺序早于成员变量初始化。

    2. 接口、抽象类不能直接用 @Slf4j

    接口不能有实例静态成员变量，注解放接口编译报错；
    抽象类可以使用。

    3. 内部类使用注意

    内部类不会共用外部类的 log，内部类要打印日志必须自己加 @Slf4j 或手动声明 Logger。

    4. 不要方法内 new Logger

    频繁创建 Logger 开销大，@Slf4j 静态常量设计就是为了单例复用。

MyBatis逆向生成

  • 插件依赖
    <build>
    <plugins>
        <!-- MyBatis 逆向生成插件 -->
        <plugin>
            <groupId>org.mybatis.generator</groupId>
            <artifactId>mybatis-generator-maven-plugin</artifactId>
            <version>1.4.2</version>
            <dependencies>
                <!-- 数据库驱动，版本和项目保持一致 -->
                <dependency>
                    <groupId>mysql</groupId>
                    <artifactId>mysql-connector-java</artifactId>
                    <version>8.0.30</version>
                </dependency>
            </dependencies>
            <configuration>
                <!-- 配置文件路径 -->
                <configurationFile>${basedir}/src/main/resources/generator/generatorConfig.xml</configurationFile>
                <!-- 覆盖已有文件 -->
                <overwrite>true</overwrite>
                <verbose>true</verbose>
            </configuration>
        </plugin>
    </plugins>
</build>
